#1 Trade.notional stored but never multiplied in — all trades priced at unit notional High
sdevpy/montecarlo/mcpricer.py · line 177

Why it's bad: Trade.__init__ accepts and stores self.notional = kwargs.get('notional', 1.0), but MonteCarloPricer.build() at line 177 aggregates with: 

results['pv'] = np.asarray(leg_cf_pvs).sum(axis=1).sum()

The sum never involves trade.notional. Any book with non-unit notionals — which is every real trade — produces prices that are exactly wrong by the notional factor. Because the default notional is 1.0, the bug is invisible in unit tests but breaks every production pricing call silently.

Fix:

results['pv'] = np.asarray(leg_cf_pvs).sum(axis=1).sum()
results['pv'] = trade.notional * np.asarray(leg_cf_pvs).sum(axis=1).sum()

#4 dump_data() calls self.params.tolist() but params starts as None Medium
sdevpy/volatility/impliedvol/models/tssvi1.py · line 152  ·  tssvi2.py · line 148  ·  logmix.py · line 366

Why it's bad: ParametricImpliedVol.__init__ sets self.params = None. If dump_data() is called before update_params() — e.g. during serialization of a freshly constructed model that has not yet been calibrated — the call chain crashes with AttributeError: 'NoneType' object has no attribute 'tolist'. While update_params now correctly coerces to ndarray, the gap between construction and first call to update_params is unguarded.

Fix: Either initialise params to an empty array in the base class, or guard in each dump_data:

def dump_data(self) -> dict:
    if self.params is None:
        raise RuntimeError("Model has no parameters yet; call update_params() first")
    return {'type': 'TsSvi1', 'params': self.params.tolist()}
#5 Maximum and BarrierDown are incomplete WIP stubs exported in the public API Medium
sdevpy/montecarlo/payoffs/exotics.py · lines 71–96

Why it's bad: Both classes are tagged "Warning: work in progress" and both raise NotImplementedError in evaluate(). They are importable from the public sdevpy.montecarlo.payoffs.exotics namespace. Maximum.__init__ sets self.eventdates at construction time (the wrong lifecycle phase — this should be deferred to set_valuation_date). BarrierDown stores a raw integer index instead of a name, bypassing the name-resolution system entirely. Any caller who accidentally instantiates either class gets a runtime error deep inside the pricing engine with no indication of why.

Fix: Until fully implemented, prefix with an underscore (_Maximum, _BarrierDown) so they are not part of the public API, or add a prominent raise NotImplementedError in __init__ itself so the failure is immediate and clearly labelled.

#6 No tests for TsIvCalibrator.calibrate, calibrate_lv_interp, or the shift_forward=True path Medium
sdevpy/tests/ · codebase-wide

Why it's bad: The end-to-end calibration pipeline (TsIvCalibrator.calibrate → prepare_target_data → TsIvObjectiveBuilder.objective) is only exercised by the notebook, never by pytest. Similarly, calibrate_lv_interp (the Dupire path) and the density-shifting branch (config.shift_forward = True, which is the default value in PdeConfig) have no automated tests. #7 PDE matrix assembly uses pure Python for loops — easily vectorized with NumPy slices Medium
sdevpy/pde/pdeschemes.py · lines 56–65, 73–80 (ThetaScheme)  ·  lines 103–113 (ImplicitScheme)

Why it's bad: Both ImplicitScheme.roll_forward and ThetaScheme.roll_forward assemble the tridiagonal band vectors using Python for j in range(n_x) loops with conditional branches. For a 250-point grid and 50 time steps, this is ~12,500 Python loop iterations per maturity call. Compared to vectorized NumPy operations (which drop to C), this is 10–100× slower than necessary.

Fix (ImplicitScheme example):

# Current (slow):
for j in range(n_x):
    main[j] = (1.0 + b * lv[j]**2)
    if j < n_x - 1:
        upper[j] = -a * lv[j + 1]**2
    if j > 0:
        lower[j - 1] = -c * lv[j - 1]**2

# Vectorized replacement:
main  = 1.0 + b * lv**2
upper = -a * lv[1:]**2
lower = -c * lv[:-1]**2
#8 black.implied_vol uses Brent with coarse xtol=1e-4 — limits calibration precision to ~1 bp Medium
sdevpy/analytics/black.py · line 30

Why it's bad: implied_vol calls minimize_scalar(..., options={'xtol': 1e-4}). The tolerance is on the vol argument directly, so the returned vol may be off by up to 1bp (100bps of a percentage point). This function is the endpoint of optionsurface.convert_option() — used during vol-to-price and price-to-vol conversions in calibration. A calibrator with 1e-10 solver tolerance feeding on targets that are only accurate to 1e-4 will plateau early. Additionally, xmax = 1.0 (100% cap) silently fails for high-vol or short-dated options that need vols above 100%. A vectorized Newton-Raphson implementation (implied_vol_newton) already exists in the same file and is both faster and more accurate.

Fix: Replace the Brent call with implied_vol_newton, or at minimum tighten xtol from 1e-4 to 1e-8 and raise xmax to 10.0.

#9 PdeConfig.shift_forward defaults to True but the function docstring says "We are not using this for now" Low
sdevpy/pde/pdeschemes.py · line 20  ·  sdevpy/pde/forwardpde.py · lines 102–103, 171

Why it's bad: PdeConfig declares shift_forward: bool = True, meaning any caller who constructs a PdeConfig without specifying this field runs the forward-shift correction. But shift_forward() at line 102 carries the docstring "Shift the density to match the forward. We are not using this for now." And price_vanillas at line 171 explicitly overrides to shift_forward=False. The inconsistency between default-on, the "not using" docstring, and the explicit override in the only production caller is confusing: a new contributor creating a PdeConfig() will silently run a code path the author labelled as unused.

Fix: Change the default to False to match actual usage, and update the docstring to describe what the function does rather than whether it is in use.

#10 New variance swaps (start ≥ valdate) always crash at pricing — current_fixing never set Low
sdevpy/montecarlo/payoffs/basic.py · lines 452, 460–463

Why it's bad: Variance.__init__ sets self.current_fixing = None. In set_valuation_date, if start ≥ valdate (a brand-new variance swap), all dates fall into self.eventdates and hist_fixing_dates = []. If get_fixings returns None or an empty list for an empty input, the condition len(hist_fixings) ≥ 1 is never met, so current_fixing stays None. Then evaluate(): 

if self.current_fixing is None:
    raise ValueError(f"Fixing not set for variance on {self.name}")

raises on every pricing call. The existing test_mc.py uses start_date = 2025-11-15 and valdate = 2025-12-15, which always has historical fixings — the failing case (start = valdate) is never covered by tests.

Fix: When hist_fixing_dates is empty (new contract), set current_fixing to the current spot from market data, or expose a set_current_fixing(spot) method and document that it must be called before pricing.
