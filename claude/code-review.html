#1 fixings.py calls Path.replace(str, str) — TypeError whenever duplicates are found High
sdevpy/market/fixings.py · line 94

Why it's bad: data_file() returns a pathlib.Path. Path.replace(target) is the OS rename/move call — it takes exactly one argument (the destination path). Calling it with two string arguments raises TypeError: replace() takes 2 positional arguments but 3 were given the moment any CSV file contains duplicate dates. The error path that is supposed to detect and isolate bad data instead crashes itself, producing a confusing traceback that points away from the actual problem. 

dupe_file = file.replace(".csv", ".dupes.csv")  # TypeError — Path.replace takes 1 arg

Fix:

dupe_file = file.replace(".csv", ".dupes.csv")
dupe_file = file.with_name(file.stem + ".dupes.csv")
#2 SabrGenerator.calibrate unpacks a single object as a 2-tuple — unconditionally broken High
sdevpy/volatility/mlsurfacegen/sabrgenerator.py · line 389

Why it's bad: MultiOptimizer.minimize (maths/optimization.py:135) returns a single scipy.OptimizeResult object. SabrGenerator.calibrate unpacks it as a two-tuple: 

result, nfev = optim.minimize(sabr_obj, x0=init_point, args=args, bounds=bounds)

This raises ValueError: too many values to unpack on every single call to calibrate(). The method is entirely unusable. The value nfev is already available as result.nfev.

Fix:

result, nfev = optim.minimize(sabr_obj, x0=init_point, args=args, bounds=bounds)
result = optim.minimize(sabr_obj, x0=init_point, args=args, bounds=bounds)
nfev = result.nfev
#3 Both production file-providers import from sdevpy.tests — breaks installed-package usage Medium
sdevpy/market/fileprovider.py · line 16  ·  sdevpy/calibration/fileprovider.py · line 5

Why it's bad: Both production classes use from sdevpy.tests import conftest to fall back to the test data directory when no root is provided. If sdevpy is installed from a wheel (standard for any published package), the tests/ subtree is absent and any code constructing a file-provider without an explicit root crashes with ModuleNotFoundError at import time. It also means a no-arg provider silently resolves to test-data paths in production. 

from sdevpy.tests import conftest  # production module depending on test layer
...
self.root = (Path(root) if root is not None else conftest.marketdata_path())

Fix: Require root explicitly, or resolve it from an environment variable / application-level config: 

self.root = (Path(root) if root is not None else conftest.marketdata_path())
if root is None:
    raise ValueError("root path must be provided explicitly")
self.root = Path(root)
#4 np.seterr(divide='raise') not restored on exception — global numpy state corrupted Medium
sdevpy/volatility/mlsurfacegen/sabrgenerator.py · lines 104, 171

Why it's bad: generate_samples_inverse sets the global numpy floating-point error mode to 'raise' at line 104 and resets it at line 171 — but only on the success path. If any exception is raised in between (e.g. inside price_straddles_ref), the reset is skipped. All subsequent numpy operations in the process — in every unrelated module — will then raise on divide-by-zero instead of returning inf, producing impossible-to-trace failures. 

np.seterr(divide='raise')      # line 104 — set
# ... exception possible here ...
np.seterr(divide='warn')       # line 171 — never reached if exception thrown

old = np.seterr(divide='raise')
try:
    # ... body ...
finally:
    np.seterr(**old)
#5 calib_black_from_vols takes square root of potentially negative variance — silent nan Medium
sdevpy/volatility/localvol/black_calib.py · line 55

Why it's bad: The Dupire forward variance vole² × te − vols² × ts can be negative when implied vols decrease with maturity (common in rates markets). np.sqrt of a negative number returns nan silently, propagating through the entire calibrated local-vol surface without any diagnostic: 

lvols.append(np.sqrt((vole**2 * te - vols**2 * ts) / (te - ts)))

Fix: Guard and log before the square root:

fwd_var = (vole**2 * te - vols**2 * ts) / (te - ts)
if fwd_var < 0.0:
    log.warning(f"Negative forward variance at T={te:.3f}; flooring to 0")
    fwd_var = 0.0
lvols.append(np.sqrt(fwd_var))
#6 Test ticker names hardcoded as module-level production data in calibration/provider.py Medium
sdevpy/calibration/provider.py · line 31

Why it's bad: A module-level dict bakes test instrument names into production code: 

name_model_map = {'ABC': 'BiExp', 'KLM': 'VSVI', 'XYZ': 'Matrix'}

ABC, KLM, and XYZ are the test fixtures from sdevpy/tests/data/. get_localvol_or_new uses this map to infer a model when model_name=None is passed — any real instrument not in this dict results in a KeyError. Production code should not contain test instrument names.

Fix: Remove the map from the module and require callers to always supply a model_name, or inject the map through a configuration mechanism.

#7 implied_vol_newton silently returns a non-converged result Medium
sdevpy/analytics/black.py · lines 65–80

Why it's bad: When max_iter is exhausted without meeting the convergence tolerance, the Newton solver returns whatever approximation it reached last — no warning, no flag, no exception. A calibration routine will receive a plausible-looking but wrong volatility and proceed as if convergence occurred. The downstream effect can be a fitted model that has absorbed solver error as parameter noise.

Fix: Track convergence and emit a warning when the loop exits early:

converged = False
for _ in range(max_iter):
    # ... Newton step ...
    if all_converged:
        converged = True; break
if not converged:
    log.warning("implied_vol_newton: max_iter reached without full convergence")

#9 MersenneTwiser — public API class name has a typo; also # pragma: no cov is invalid Low
sdevpy/maths/rand/rng.py · lines 39, 64

Why it's bad: The class is named MersenneTwiser (missing a 't'). Because it is exported as a public API symbol, a rename requires a deprecation alias to avoid breaking callers — and the longer this stays, the more callers accumulate. Separately, line 39 has # pragma: no cov, which is not a recognised coverage pragma. The correct directive is # pragma: no cover. The current typo means the abstract method is not actually excluded from coverage reports, silently reducing measured coverage. 

class MersenneTwiser(RandomNumberGenerator): ...    # line 64
    pass  # pragma: no cov                           # line 39

class MersenneTwister(RandomNumberGenerator): ...
    pass  # pragma: no cover

MersenneTwiser = MersenneTwister  # backward-compatible alias
