#1 Variance always crashes for new contracts — current_fixing never set when start ≥ valdate High
sdevpy/montecarlo/payoffs/basic.py · lines 452, 460–463

Why it's bad: Variance.__init__ sets self.current_fixing = None. In set_valuation_date, when start ≥ valdate (a brand-new variance swap with no historical fixings), hist_fixing_dates is empty and the condition len(hist_fixings) ≥ 1 is never satisfied, so current_fixing stays None. Then evaluate(): 

if self.current_fixing is None:
    raise ValueError(f"Fixing not set for variance on {self.name}")

raises on every pricing call. The existing test in test_mc.py uses start_date = 2025-11-15 with valdate = 2025-12-15, so there are always historical fixings — the new-contract path is never exercised by the test suite.

Fix: When hist_fixing_dates is empty, initialise current_fixing to the current spot from market data, or expose a set_current_fixing(spot) method and document that it must be called before pricing.

#2 ThetaScheme.roll_forward y-vector still built with a Python loop — first half not yet vectorized Medium
sdevpy/pde/pdeschemes.py · lines 56–65

Why it's bad: The band-matrix assembly for ThetaScheme (lines 74–77) was vectorized this cycle, but the preceding y-vector construction — which builds the right-hand side of the linear system — remains a Python loop with conditional branches. This runs at Python speed for every time step: 

# Current (slow) — lines 56–65
for j in range(n_x):
    p_tmp = (1.0 - one_m_theta_dt_2 * b * lv[j]**2) * p[j]
    if j < n_x - 1:
        p_tmp += one_m_theta_dt_2 * a * lv[j + 1]**2 * p[j + 1]
    if j > 0:
        p_tmp += one_m_theta_dt_2 * c * lv[j - 1]**2 * p[j - 1]
    y[j] = p_tmp

Fix — replace with NumPy slices matching the band-matrix pattern immediately below:

y      = (1.0 - one_m_theta_dt_2 * b * lv**2) * p
y[:-1] += one_m_theta_dt_2 * a * lv[1:]**2  * p[1:]
y[1:]  += one_m_theta_dt_2 * c * lv[:-1]**2 * p[:-1]
#3 ExplicitScheme.roll_forward is entirely a Python loop — no vectorization applied Medium
sdevpy/pde/pdeschemes.py · lines 142–153

Why it's bad: While ImplicitScheme was fully vectorized this cycle, ExplicitScheme retains the original pure-Python loop structure. For a 250-point grid and 50 time steps this is ~12,500 Python loop iterations per pricing call, compared to the three NumPy slice operations that would replace it. Although the Rannacher scheme delegates to ThetaScheme(0.5) after the initial implicit steps (not to ExplicitScheme), the class remains in the public API.

Fix — vectorize matching the ImplicitScheme pattern:

y      = (1.0 - b * lv**2) * p
y[:-1] += a * lv[1:]**2  * p[1:]
y[1:]  += c * lv[:-1]**2 * p[:-1]
#4 Maximum and BarrierDown are incomplete WIP stubs exported in the public API Medium
sdevpy/montecarlo/payoffs/exotics.py · lines 71–96

Why it's bad: Both classes carry a "Warning: work in progress" docstring and raise NotImplementedError in evaluate(), yet they are importable from the public sdevpy.montecarlo.payoffs.exotics namespace. Maximum.__init__ sets self.eventdates at construction time — violating the payoff lifecycle contract where this belongs in set_valuation_date. Any caller who instantiates either class gets a NotImplementedError deep inside the pricing engine with no clear indication of why.

#6 black.implied_vol uses Brent with xtol=1e-4 — limits precision to ~1 bp Medium
sdevpy/analytics/black.py · line 30

Why it's bad: implied_vol calls minimize_scalar(..., options={'xtol': 1e-4, 'maxiter': 100}). The tolerance is on the vol argument directly, so the returned vol may be wrong by up to 1 bp. This function is used during vol-to-price and price-to-vol conversions in calibration: a solver set to tol=1e-10 fed targets accurate only to 1e-4 will plateau early. Additionally, xmax = 1.0 silently caps at 100%, which fails for short-dated or high-vol options. A vectorized Newton-Raphson implementation (implied_vol_newton) already exists in the same file and is both faster and more accurate.

Fix: Replace the Brent call with implied_vol_newton, or at minimum tighten xtol from 1e-4 to 1e-8 and raise xmax to 10.0.

#10 [[None]] * n_times creates aliased inner lists — misleading pattern even though it is safe Low
sdevpy/volatility/localvol/dupire_calib.py · lines 153–154

Why it's bad: [[None]] * n_times creates a list where every slot holds a reference to the same inner [None] object. In the actual code each slot is immediately replaced by whole-slot assignment (lv[i] = dupire_formula(...)), so the aliasing is never exploited and the code is correct. But any reader unfamiliar with the Python aliasing rule will spend time verifying there is no hidden mutation bug. The same pattern appears on line 154 for moneynesses.

Fix: Use a flat list to make intent unambiguous: 

lv          = [[None]] * n_times
moneynesses = [[None]] * n_times
lv          = [None] * n_times
moneynesses = [None] * n_times

Since each slot is overwritten by assignment, the inner list serves no purpose.
