Provider pattern landed — but two regressions in mcpricer.py and a Protocol gap need immediate attention

The MarketDataProvider Protocol is now implemented and wired into the book, payoffs, calibrator, and tests — a significant architecture improvement. test_tssvi1_calibrate closes the biggest testing gap from the 9th review. The PDE schemes are fully vectorized and shift_forward now defaults to False. However, the provider refactor left price_vanilla_surface broken (wrong call signature), and get_spots is called on MarketDataProvider-typed objects but is absent from the Protocol, undermining type safety. The variance new-contract crash and the two WIP stubs in the public namespace remain unresolved from the 9th review.

Fixed since 9th review: MarketDataProvider Protocol implemented in provider.py  ·  all market-data tests updated to use MarketDataFileProvider  ·  test_tssvi1_calibrate added — full calibration round-trip now covered  ·  PDE schemes vectorized (was loop-based)  ·  PdeConfig.shift_forward now defaults to False, matching actual usage  ·  get_forward_curves moved to provider.py as a free function on the interface

#1 price_vanilla_surface crashes immediately — missing md argument after provider refactor High
sdevpy/montecarlo/mcpricer.py · line 221

Why it's bad: price_book was refactored to take md: MarketDataProvider as its third positional argument, but price_vanilla_surface was never updated. At line 221 it calls: 

sim_prices = price_book(valdate, book, lv_map={name: lv}, **kwargs)

lv_map={name: lv} lands in **kwargs, leaving md unbound. Python will raise TypeError: price_book() missing 1 required positional argument: 'md' on every call. Additionally, line 225 still calls the module-level get_yieldcurve directly rather than going through a provider. This function is effectively dead code that silently rotted during the refactor.

Fix: Add an md: MarketDataProvider parameter to price_vanilla_surface and thread it through to price_book and the discount curve retrieval:

def price_vanilla_surface(valdate, expiries, strikes, name, lv, md: MarketDataProvider, **kwargs):
    ...
    sim_prices = price_book(valdate, book, md, lv_map={name: lv}, **kwargs)
    disc_curve = md.get_yieldcurve(book.csa_curve_id, valdate)
#2 get_spots absent from MarketDataProvider Protocol but called on Protocol-typed objects High
sdevpy/montecarlo/mcpricer.py · line 39  ·  sdevpy/market/provider.py · line 92

Why it's bad: The Protocol declares get_spot (singular) but not get_spots (plural). Both price_book and get_forward_curves call md.get_spots(names, valdate) / provider.get_spots(names, valdate) on objects typed as MarketDataProvider. mypy reports an error on both sites. More importantly, any alternative provider implementation (Bloomberg, database) that correctly implements every method in the Protocol will still fail at runtime because get_spots is missing from the declared contract.

Fix: Add get_spots to the Protocol:

def get_spots(self, names: list[str], valdate: dt.datetime) -> np.ndarray: ...
#3 get_fixings annotated as -> FixingHandler in the Protocol but returns values Medium
sdevpy/market/provider.py · line 22 (Protocol)  ·  lines 42–45 (implementation)

Why it's bad: The Protocol declares: 

def get_fixings(self, name, dates, **kwargs) -> FixingHandler: ...

But the implementation at lines 42–45 returns handler.value(dates), which is a float or list[float] depending on the input. The callers in Average.set_valuation_date and Variance.set_valuation_date (basic.py lines 270 and 494) treat the return as a list of floats and call np.asarray(...).sum() on it — which would crash if the annotation were accurate and a real FixingHandler were returned.

Fix: Correct the Protocol annotation to match reality:

def get_fixings(self, name: str,
               dates: dt.datetime | list[dt.datetime],
               **kwargs) -> float | list[float]: ...
#4 New variance swap (start ≥ valdate) always crashes at pricing — current_fixing never set Medium
sdevpy/montecarlo/payoffs/basic.py · lines 454, 499–504

Why it's bad: Variance.__init__ sets self.current_fixing = None. In set_valuation_date, when start ≥ valdate (a brand-new contract), hist_fixing_dates = []. Then md.get_fixings(name, []) returns [] and the guard at line 503 (len(hist_fixings) ≥ 1) is never satisfied, so current_fixing stays None. Every subsequent evaluate() call raises: 

if self.current_fixing is None:
    raise ValueError(f"Fixing not set for variance on {self.name}")

The existing test_mc.py always uses start = 2025-11-15 (one month before valdate), so historical fixings always exist and the bug is invisible. Any real new-issue trade (start = valdate) will crash.

Fix: When no historical fixings exist, initialise current_fixing from the spot on valdate:

if len(hist_fixings) == 0:
    self.current_fixing = md.get_spot(self.name, valdate)
elif len(hist_fixings) >= 1:
    self.current_fixing = hist_fixings[-1]
#5 Maximum and BarrierDown are WIP stubs publicly exported Medium
sdevpy/montecarlo/payoffs/exotics.py · lines 71–96

Why it's bad: Both classes are tagged "Warning: work in progress" and raise NotImplementedError in evaluate(). They are importable from sdevpy.montecarlo.payoffs.exotics. Maximum also sets self.eventdates in __init__ (the wrong lifecycle phase — should be deferred to set_valuation_date), and BarrierDown takes a raw integer index rather than a name, bypassing the name-resolution system. A caller who accidentally instantiates either class gets an error deep inside the pricing engine with no indication of why it fails immediately.

Fix: Until fully implemented, prefix with underscore to remove from the public API, or raise NotImplementedError in __init__:

class _Maximum(Payoff):   # not exported
    ...

# or, to fail at construction time:
def __init__(self, left, right):
    raise NotImplementedError("Maximum is not yet implemented")
#6 LocalVol.dump_data family crashes if valdate is not supplied (the default) Medium
sdevpy/volatility/localvol/localvol.py · lines 329–332 (ConstantLocalVol)  ·  lines 142–144 (TimeInterpolatedLocalVol)

Why it's bad: LocalVol.__init__ defaults valdate to None via kwargs.get('valdate', None). Both dump_data implementations then call self.valdate.strftime(...) unconditionally. Any call to dump_data() on an LV object that was constructed without an explicit valdate — including every PDE test that uses ConstantLocalVol(0.20) — raises AttributeError: 'NoneType' object has no attribute 'strftime'. The test suite avoids this only by never calling dump_data on these objects.

Fix: Guard the serialization or require valdate explicitly:

def dump_data(self) -> dict:
    if self.valdate is None:
        raise RuntimeError("Cannot serialize LocalVol without a valdate")
    return {'name': self.name, 'valdate': self.valdate.strftime(dates.DATE_FORMAT), ...}
#7 TsIvCalibrator.calibrate does not check result.success — silent convergence failure Medium
sdevpy/volatility/impliedvol/impliedvol_calib.py · lines 86–91

Why it's bad: After calling optimizer.minimize, the calibrator assigns self.sol = self.result.x and forces the model to that point — regardless of whether the optimizer converged. scipy.optimize.OptimizeResult.success is never inspected. A failed calibration (hit max iterations, ill-conditioned surface, infeasible bounds) will silently leave the model at a non-converged parameter set, which is then used for pricing and further LV calibration. In production, this produces subtly wrong prices with no warning.

Fix: Raise or at minimum warn on failed convergence:

if not self.result.success:
    import warnings
    warnings.warn(
        f"Calibration did not converge: {self.result.message}",
        RuntimeWarning, stacklevel=2
    )
self.model.update_params(self.sol)
#8 provider.py __main__ block calls non-existent method get_yield_curve Low
sdevpy/market/provider.py · line 109

Why it's bad: The if __name__ == "__main__" block calls md_provider.get_yield_curve(...) but the actual method is named get_yieldcurve (no underscore). Running the file directly produces AttributeError: 'MarketDataFileProvider' object has no attribute 'get_yield_curve'. This is a documentation and discoverability problem: developers exploring the provider API by running the file get an immediate crash. The Protocol itself has the same method named get_yieldcurve, so the mismatch is in the example code only. 

obj = md_provider.get_yield_curve("USD.SOFR.1D", valdate)
obj = md_provider.get_yieldcurve("USD.SOFR.1D", valdate)
