#2 Inconsistent infeasibility penalty across objective builders High
sdevpy/volatility/localvol/localvol_calib.py : 189  ·  sdevpy/volatility/impliedvol/impliedvol_calib.py : 52–53

The two objective builders handle infeasible parameters differently. LvObjectiveBuilder returns self.cf_prices.sum() (line 189 of localvol_calib.py) — a value that varies with expiry and number of strikes. TsIvObjectiveBuilder returns constants.FLOAT_INFTY (line 53 of impliedvol_calib.py), but the comment above it reads "Didn't this cause problems before?", indicating the approach is uncertain and has not been systematically tested.

Why this matters: an optimizer that receives cf_prices.sum() as penalty will see different "wall heights" at different expiries. The optimizer may steer toward expiries where the penalty happens to be smaller, rather than toward feasibility. FLOAT_INFTY is the correct sentinel — but the uncertainty comment in impliedvol_calib.py means neither builder is conclusively right. The two builders should align on the same strategy, and the strategy should be validated. 

# localvol_calib.py:189 — scale-dependent
-            return self.cf_prices.sum()
+            return constants.FLOAT_INFTY

# impliedvol_calib.py:53 — uncertain comment should be resolved
-            return constants.FLOAT_INFTY  # Didn't this cause problems before?
+            return constants.FLOAT_INFTY  # Validated: works correctly with SLSQP/L-BFGS-B
#3 svi() returns a NaN array on invalid parameters Medium
sdevpy/volatility/impliedvol/models/svi.py : 40–42

When svi_check_params() returns infeasible, svi() returns np.full_like(x, np.nan) (line 41) instead of raising an exception. This is inconsistent with every other model in the codebase: TsSvi1.calculate() returns np.full_like(k, np.nan), LogMix raises RuntimeError for missing params, and svi_check_params itself returns FLOAT_INFTY as a penalty — signalling that infinity, not NaN, is the agreed sentinel. Callers that do not check for NaN (e.g. any downstream RMSE calculation) will silently propagate NaN through the optimizer without signalling a problem. The commented-out alternative on line 42 (# raise ValueError(...)) suggests this was a deliberate but unevaluated trade-off. 

-        return np.full_like(np.asarray(x, dtype=float), np.nan)
-        # raise ValueError("Invalid SVI parameters")
+        raise ValueError(f"Invalid SVI parameters: check_params failed")

Note: svi_formula() delegates directly to svi(), and SviSection.value() delegates to svi_formula() — so NaN can reach the PDE or calibration objective without any intermediate warning.

#5 check_consistency() is dead code with a "not used yet" comment Medium
sdevpy/volatility/impliedvol/impliedvol_calib.py : 125–139

TsIvCalibrator.check_consistency() is a 15-line method that the docstring explicitly labels "not used yet, but will be used in a later phase". It is never called from anywhere in the codebase. Shipping untested, dead code under a "coming soon" label has two costs: (1) it creates a false impression that negative-rate handling is implemented; (2) when eventually called, the method may be silently out-of-date with the surrounding API (e.g. OptionTarget schema changes).

Fix: Delete the method. Implement it in a separate commit when it is actually needed and testable.

#7 Duplicate validation in TsSvi1.smile_parameters() Medium
sdevpy/volatility/impliedvol/models/tssvi1.py : 50–54 vs 108–114

check_global_params() validates both r (lines 108–109) and delta + 1 (lines 112–113), returning (False, FLOAT_INFTY) on failure. calculate() calls check_global_params() before ever reaching smile_parameters(). Despite this, smile_parameters() repeats both checks and raises ValueError (lines 50–54). The ValueError guards in smile_parameters() will never fire through the normal code path, but will throw exceptions when smile_parameters() is called directly (e.g. from check_params() via sample_times), making the error-handling contract inconsistent.

Fix: Remove lines 50–54 from smile_parameters(). check_global_params() is the single source of truth.

#8 formula parameter has no type annotation Medium
sdevpy/volatility/localvol/localvol.py : 43

ParamLocalVolSection.__init__(self, time: float, formula) — the formula callable is the central dependency of the section (it supplies the local vol function used at every PDE step) but carries no type hint. Every subclass constructor inherits this gap. Without an annotation, static type-checkers cannot verify that subclass implementations pass a compatible callable, and readers must trace all call sites to infer the expected signature (t, x, params) -> ArrayLike.
