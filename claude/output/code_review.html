#1 New-contract Variance: first log-return is silently skipped yet counted in denominator High
sdevpy/montecarlo/payoffs/basic.py · lines 487–496

Why it's bad: When start ≥ valdate (a brand-new swap), hist_fixing_dates is empty, get_fixings returns an empty list, and the condition len(hist_fixings) ≥ 1 is never met — so self.current_fixing stays None. In evaluate() the guarded block is skipped: 

if self.current_fixing is not None:   # False for a new contract → skipped
    var_sum = var_sum + np.power(np.log(spot_paths[:, 0] / self.current_fixing), 2)

The first log-return — from the current spot at valdate to the first scheduled counting date — is never added to var_sum. Yet self.n_returns was computed as len(alldates) - 1, which includes that return in the denominator. The realised variance is therefore understated, with the error growing as the first return becomes a larger fraction of the total. The existing test always uses start_date = 2025-11-15 with valdate = 2025-12-15, so there are always historical fixings and the branch is never exercised.

Fix: When hist_fixing_dates is empty, seed current_fixing from the current spot:

if hist_fixings is not None and len(hist_fixings) >= 1:
    self.current_fixing = hist_fixings[-1]
else:
    self.current_fixing = md.get_spot(self.name, valdate)
#2 Sqrt.evaluate has no zero-floor — negative paths produce silent nan High
sdevpy/montecarlo/payoffs/basic.py · line 380

Why it's bad: Variance is theoretically non-negative (sum of squared log-returns), but floating-point noise on near-zero paths can produce tiny negatives. np.sqrt of a negative number returns nan silently: 

def evaluate(self, mkt_state: dict):
    return np.sqrt(self.subpayoff.evaluate(mkt_state))

A single nan path propagates through every subsequent algebra node (Sub, Mul, discounting) without raising, so the final PV is contaminated silently. Any vol-swap book with near-zero realised variance on some paths will produce wrong numbers with no error signal.

Fix:

def evaluate(self, mkt_state: dict):
    return np.sqrt(np.maximum(self.subpayoff.evaluate(mkt_state), 0.0))
#3 VectorLocalVol.refresh_sections calls super().__init__, silently resetting valdate, name, snapdate High
sdevpy/volatility/localvol/localvol.py · lines 300–312

Why it's bad: refresh_sections is meant to resync the internal sections list after modifying self.vol_grid. It does so by calling super().__init__(sections, **kwargs) which chains up through TimeInterpolatedLocalVol.__init__ to LocalVol.__init__(**kwargs). If called with no arguments (the common case), that base call sets: 

self.valdate  = kwargs.get('valdate',  None)     # → None
self.name     = kwargs.get('name',    'Unknown') # → 'Unknown'
self.snapdate = kwargs.get('snapdate', self.valdate) # → None

A subsequent dump_data() call would then raise: "Cannot dump ConstantLocalVol: no valuation date" — even though the object was correctly constructed with a valdate. The bug is latent: it bites whenever a caller rebuilds the vol grid and then serializes or queries valdate.

Fix: Replace the super().__init__ call with a targeted section rebuild that does not touch the base-class fields: 

def refresh_sections(self, **kwargs) -> None:
    if len(self.vol_grid) != len(self.t_grid):
        raise ValueError(...)
    self.sections = [FlatLocalVolSection(t, v)
                     for t, v in zip(self.t_grid, self.vol_grid)]
#4 dump_data() crashes with AttributeError if called before update_params() Medium
sdevpy/volatility/impliedvol/parametric_impliedvol.py · line 10  ·  tssvi1.py · line ~152  ·  tssvi2.py · line ~148  ·  logmix.py · line ~366

Why it's bad: ParametricImpliedVol.__init__ sets self.params = None. Every concrete dump_data() implementation calls self.params.tolist() without guarding against this initial state. Any serialization of a freshly constructed model before calibration raises: 

AttributeError: 'NoneType' object has no attribute 'tolist'

Fix: Guard at the base class level so all subclasses inherit the check: 

def update_params(self, x):
    self.params = np.asarray(x, dtype=float)

def _require_params(self):
    if self.params is None:
        raise RuntimeError("Model has no parameters; call update_params() first")

Then call self._require_params() at the top of each concrete dump_data().

#5 implied_vol_newton silently returns unconverged result when max_iter is reached Medium
sdevpy/analytics/black.py · lines 59–85

Why it's bad: Newton-Raphson exits the loop when either np.all(np.abs(diff) < tol) is true or when max_iter iterations are exhausted. In the second case the function returns the last iterate without any indication of failure. Near-zero vega (deep OTM, very short dated) can cause Newton to diverge or stall; the caller has no way to detect this and will silently use a bad implied vol in calibration or pricing.

Fix: Track convergence and warn:

for _ in range(max_iter):
    ...
    if np.all(np.abs(diff) < tol):
        return vol
log.warning("implied_vol_newton did not converge")
return vol
#6 calib_black_from_vols passes potentially negative argument to np.sqrt Medium
sdevpy/volatility/localvol/black_calib.py · line 55

Why it's bad: The Dupire-bootstrap formula for piecewise-constant local vol is: 

lvols.append(np.sqrt((vole**2 * te - vols**2 * ts) / (te - ts)))

If the implied vol term structure decreases fast enough that vole² · te < vols² · ts (e.g., a steep inversion in the term structure), the argument to np.sqrt is negative and returns nan silently. The resulting VectorLocalVol contains a nan section and all subsequent PDE or MC pricing will produce nan with no indication of where it came from.

Fix: Guard with a meaningful error before the sqrt:

forward_var = (vole**2 * te - vols**2 * ts) / (te - ts)
if forward_var < 0:
    raise ValueError(f"Negative forward variance at pillar {i}: inverted vol term structure")
lvols.append(np.sqrt(forward_var))
#7 Dupire calibration path (dupire_calib.py, lvsection_calib.py) has no automated tests Medium
sdevpy/volatility/localvol/dupire_calib.py  ·  sdevpy/volatility/localvol/lvsection_calib.py

Why it's bad: The TsSvi1 calibration round-trip test added this cycle is a meaningful step forward, and the shift_forward path is now exercised. However, the Dupire density-fitting calibration (dupire_calib.py) and the per-section LV calibration (lvsection_calib.py) remain entirely outside automated test coverage. These are the paths used to produce the stored LV files that the MC and PDE pricers consume. #9 Dead np.zeros allocations in ThetaScheme and ImplicitScheme immediately overwritten Low
sdevpy/pde/pdeschemes.py · lines 63–65, 91–93

Why it's bad: The vectorisation of the PDE band-vector assembly is correct and fast, but the previous loop scaffolding left orphaned allocations: 

upper = np.zeros(n_x - 1)   # allocated …
main  = np.zeros(n_x)        # allocated …
lower = np.zeros(n_x - 1)   # allocated …
# Vectorized
main  = 1.0 + theta_dt_2 * b * lv**2        # … and immediately replaced
upper = -theta_dt_2 * a * lv[1:]**2
lower = -theta_dt_2 * c * lv[:-1]**2

The three np.zeros calls allocate memory that is discarded on the very next line. Each call to roll_forward performs 3 pointless allocations. With 250 mesh points × 50 time steps × many maturities, this accumulates.

Fix: Simply remove the three np.zeros lines. The vectorized assignments create the arrays directly.

#10 VectorLocalVol.vol_grid can silently diverge from internal sections Low
sdevpy/volatility/localvol/localvol.py · lines 280–298

Why it's bad: VectorLocalVol stores self.vol_grid as a mutable public list. The internal self.sections (inherited from TimeInterpolatedLocalVol) are built once at construction and not automatically kept in sync. A caller can modify vol_grid directly: 

lv.vol_grid[2] = 0.35   # looks like an update …
lv.value(t, logm)        # … but still returns the old value from sections[2]

The stale read is silent and hard to debug. refresh_sections() exists but is not documented as a required call after mutation, and is itself broken (see issue #3).

Fix: Make the sync automatic by overriding __setitem__ on a custom list subclass, or expose a proper update_vol(t_idx, vol) method that updates both vol_grid and sections[t_idx] atomically. At minimum, prefix vol_grid with an underscore and provide a property that raises if direct mutation is attempted.
