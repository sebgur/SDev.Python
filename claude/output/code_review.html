#1 Infeasibility penalty is scale-dependent across expiries High
sdevpy/volatility/localvol/localvol_calib.py : 189

When LvObjectiveBuilder.check_params() returns infeasible, the objective returns self.cf_prices.sum(). This sum depends on the number of strikes and their absolute price levels at the current expiry. As the outer calibration loop advances through expiries, the penalty magnitude shifts — so the optimizer sees an inconsistent cost landscape: equally infeasible parameters produce different penalty values at different expiries. This can steer the optimizer toward expiries with a lower penalty rather than toward feasibility.

The comment directly above this line notes: "Claude recommends using constants.FLOAT_INFTY. But didn't we use it before and it led to some problems and that's why we're doing this now? To be tested again." That investigation is worth completing. TsIvObjectiveBuilder (in impliedvol_calib.py:53) already uses FLOAT_INFTY; both builders should agree on the same strategy. 

-            return self.cf_prices.sum()
+            return constants.FLOAT_INFTY
#2 svi() returns a NaN array on invalid parameters Medium
sdevpy/volatility/impliedvol/models/svi.py : 40–42

When svi_check_params() returns infeasible, svi() returns np.full_like(x, np.nan). This is inconsistent with every other model in the codebase: svi_check_params() itself returns FLOAT_INFTY as a penalty; TsSvi1.calculate() returns np.full_like(k, np.nan) (same smell, but less harmful since the model is not used directly by the LV PDE loop). Any downstream RMSE calculation receiving NaN will silently propagate NaN through the optimizer without triggering a recognisable error signal. The commented-out alternative on line 42 (# raise ValueError("Invalid SVI parameters")) shows this was a deliberate trade-off that was never fully evaluated. 

-        return np.full_like(np.asarray(x, dtype=float), np.nan)
-        # raise ValueError("Invalid SVI parameters")
+        raise ValueError(f"Invalid SVI parameters: svi_check_params failed")

#4 check_consistency() is dead code with a "not used yet" comment Medium
sdevpy/volatility/impliedvol/impliedvol_calib.py : 125–139

TsIvCalibrator.check_consistency() is a 15-line method whose docstring explicitly labels it "not used yet, but will be used in a later phase". It is never called from anywhere in the codebase. Shipping untested, dead code has two costs: it creates a false impression that negative-rate handling is in place; and when eventually called, the method may be silently out-of-date with the surrounding API (e.g. OptionTarget schema changes).

Fix: Delete the method now. Implement it in a focused commit when it is needed and testable.

#6 Duplicate validation in TsSvi1.smile_parameters() Medium
sdevpy/volatility/impliedvol/models/tssvi1.py : 50–54 vs 105–111

check_global_params() validates r (lines 105–106) and delta + 1 (lines 109–110), returning (False, FLOAT_INFTY) on failure. calculate() calls check_global_params() first, so smile_parameters() is never reached with invalid r or delta through the normal path. Despite this, smile_parameters() re-validates both at lines 50–54 and raises ValueError. These guards will only fire when smile_parameters() is called directly (as happens inside check_params() via the sample-times loop), making the error-handling contract inconsistent: the same invalid input raises an exception from one entry point and returns (False, penalty) from another.

Fix: Remove lines 50–54; let check_global_params() be the single validator.
