General Advice

Add a static type checker to CI

  • Two functions already have provably wrong return-type annotations (print_python_bit_version() -> str and list_mcal_calendars() -> list[str] both return None), and the lvsection_calib int/float confusion would have been a type error at check time. A single mypy (or pyright) run on sdevpy/ with --ignore-missing-imports would surface these immediately and make future regressions of this class impossible to land.

Enforce a uniform interface contract for model classes

  • FactorModel ABC is not consistently implemented. MultiAssetHeston does not inherit from it; its initial_state has a different signature; it has no evolve_state method. Define a formal protocol with exact signatures for initial_state(), evolve_state(), and dim, and enforce it through inheritance. That turns this class of integration bug into an import-time error.
  • Implied vol model lifecycle. The params = None gap until update_params() is called is a recurring source of opaque crashes across TsSvi1, TsSvi2, and LogMix. Lift a uniform _require_params() guard into ParametricImpliedVol and call it at the top of every calculate() and check_params().

Replace bare print() with logging throughout

  • There are roughly a dozen print() calls remaining in production and utility code — in meanreversion.py:104, localvol_factory.py:171, tools.py:40, scalendar.py:208, and several if __name__ == "__main__": print("Hello") stubs scattered across production modules. Library code should never write to stdout. Use log = logging.getLogger(__name__) (already established elsewhere) for informational output, and either remove the __main__ stubs or replace them with meaningful example scripts.

Standardize the test tolerance convention

  • np.allclose(test, ref, 1e-10) sets rtol=1e-10 (the third positional argument), not atol. At ~30 call sites across the test suite the wrong tolerance axis is being controlled. Adopt np.allclose(test, ref, rtol=0.0, atol=1e-10) explicitly, or define a project-wide assert_close(a, b, atol) wrapper to make the intent unambiguous and consistently applied.

Add end-to-end calibration smoke tests

  • The calibration pipeline — TsIvCalibrator.calibrate, calibrate_lv_section, and MC pricing with the full book-building path — has zero automated test coverage. Regressions in these paths are invisible to CI. The lvsection_calib wrong-section bug found in this review, the earlier notional bug, and the eventdates.max() crash would all have been caught immediately by a single smoke test that runs the full calibration on a toy surface and checks RMSE < threshold.

Prefer context managers for temporary NumPy error state

  • sabr.py uses the global np.seterr(invalid='ignore') ... np.seterr(invalid='warn') pair. An exception between these two lines leaves the global state as 'ignore' permanently. Replace with with np.errstate(invalid='ignore'):, which is already the correct pattern in every other model file in the codebase.

#1 eventdates.max() called on a Python list — crashes every real price_book call High
sdevpy/montecarlo/mcpricer.py · line 23

Why it's bad: build_timegrid is declared as receiving eventdates: list[dt.datetime]. Python lists have no .max() method; this raises AttributeError: 'list' object has no attribute 'max' on every call to price_book — the main entry point of the MC pricer. The bug is masked only in contexts where eventdates happens to be a numpy array rather than a plain list. 

max_date = eventdates.max()         # AttributeError if eventdates is a Python list
max_date = max(eventdates)           # or: np.asarray(eventdates).max()
#2 MultiAssetHeston is architecturally incompatible with PathGenerator High
sdevpy/models/multiasset_heston.py · lines 4–37

Why it's bad: MultiAssetHeston does not inherit from FactorModel (the ABC that PathGenerator depends on). It implements step() but PathGenerator.generate_paths calls model.evolve_state() — which does not exist on this class, producing AttributeError at simulation time. Additionally, initial_state requires a positional n_paths argument, but PathGenerator calls self.model.initial_state() with no arguments. The model cannot be used with the existing engine in any configuration.

Fix: Inherit from FactorModel, implement evolve_state(state, dt, z) wrapping the existing step logic, and remove n_paths from initial_state (it can be stored internally or read from the generator context at call time).

#3 lvsection_calib passes integer loop counter to section(t: float) — wrong LV section from expiry 2 onward High
sdevpy/volatility/localvol/lvsection_calib.py · line 114

Why it's bad: section(t) in TimeInterpolatedLocalVol performs a time lookup via lower_bound(t_grid, t), where t_grid holds fractional-year pillar times like [0.25, 0.5, 1.0, 2.0, 5.0]. The calibration loop at line 103 iterates over exp_idx in range(len(expiry_grid)) (0, 1, 2…) and passes this integer directly to lv.section(exp_idx). At exp_idx=2 the lookup requests time 2.0, which maps to the 4th pillar — not the 3rd expiry. The correct method section_at_index(exp_idx) is already used correctly at lines 136 and 239 for update_params and check_params. 

bounds = lv.section(exp_idx).constraints()          # int used as float time
bounds = lv.section_at_index(exp_idx).constraints() # correct
#4 implied_vol uses Brent with bracket=, not bounds= — can return negative volatility silently High
sdevpy/analytics/black.py · line 47  ·  sdevpy/analytics/bachelier.py · line 126

Why it's bad: minimize_scalar(method='brent', bracket=(xmin, xmax)) does not constrain the search to [xmin, xmax]. The bracket parameter is only an initial bracketing hint; Brent's method can and does step outside it when the error surface is asymmetric. For noisy or out-of-model prices, the solver can return a negative volatility (e.g., -0.15) with no error raised, and the result propagates silently into calibration targets. The correct API for a bounded search is method='bounded' with bounds=(xmin, xmax). The same bug exists in bachelier.py at line 126, compounded by an xmax=1.0 that is too narrow for high-forward normal vols. 

res = minimize_scalar(fun=error, bracket=(xmin, xmax), method='brent', options=options)
res = minimize_scalar(fun=error, bounds=(xmin, xmax),  method='bounded', options=options)
#5 WorstOfBarrier only registers expiry as event date — continuous barrier is never monitored High
sdevpy/montecarlo/payoffs/exotics.py · line 46

Why it's bad: set_valuation_date sets self.eventdates = [self.expiry] — a single date. The event-path grid therefore contains only the expiry point. The barrier monitoring code in evaluate iterates over event_paths, which has shape (n_paths, 1, n_assets) — a single point-in-time check at expiry. A continuous or daily-monitored knock-out barrier never knocks on intermediate dates because those dates are not in the event grid. The barrier is effectively a European condition.

Fix: Register monitoring dates (either a daily schedule or user-supplied) as event dates in set_valuation_date, so the event-path grid contains sufficient points for path-dependent barrier detection.

#6 np.seterr(invalid='ignore') in sabr.py leaks global NumPy state on exception Medium
sdevpy/volatility/impliedvol/models/sabr.py · lines 30, 35

Why it's bad: The SABR formula suppresses NumPy invalid-operation warnings with np.seterr(invalid='ignore') and restores them at line 35. If any exception is raised inside the suppressed block (bad log, division, etc.), the restore call at line 35 is never reached and the global warning state remains 'ignore' for the rest of the process. Subsequent NaN-producing operations in unrelated code will fire silently. Every other model in the codebase uses the safe context-manager form. 

np.seterr(invalid='ignore')
correction = np.where(z2 > 10.0 * m_epsilon, z / np.log(...), 1.0 - ...)
np.seterr(invalid='warn')
with np.errstate(invalid='ignore'):
    correction = np.where(z2 > 10.0 * m_epsilon, z / np.log(...), 1.0 - ...)
#7 TsSvi1 and TsSvi2 give an opaque TypeError when params is None Medium
sdevpy/volatility/impliedvol/models/tssvi1.py · lines 33–34  ·  tssvi2.py · lines 35–36

Why it's bad: calculate() immediately calls check_global_params(), which calls self.get_parameters(self.params), which calls len(self.params). The base class ParametricImpliedVol.__init__ sets self.params = None. Calling calculate() before update_params() raises TypeError: object of type 'NoneType' has no len() — no indication that the model is simply uninitialised. LogMix already guards against this correctly. The guard should be lifted into the base class or added uniformly to both TsSvi1.check_global_params and TsSvi2.check_global_params. 

if self.params is None:
    raise RuntimeError("Model has no parameters yet; call update_params() first")
#8 print_python_bit_version() and list_mcal_calendars() are annotated to return a value but return None Medium
sdevpy/utilities/tools.py · line 38  ·  sdevpy/utilities/scalendar.py · line 207

Why it's bad: Both functions print to stdout and return nothing, but their type annotations promise a return value. Any caller expecting to use the return value silently receives None. 

def print_python_bit_version() -> str:   # declared -> str
    print(struct.calcsize("P") * 8)        # returns None

def list_mcal_calendars() -> list[str]:  # declared -> list[str]
    print(mcal.get_calendar_names())       # returns None

Fix: Replace print(...) with return ... in both functions so the computed value is actually returned to the caller.

#9 np.allclose positional third argument is rtol, not atol — wrong tolerance axis across the test suite Medium
sdevpy/tests/ · ~30+ call sites across test_analytics.py, test_mc.py, test_pde.py, test_impliedvol.py, test_localvol.py, test_localvol_calib.py, test_timegrids.py, test_yieldcurves.py, test_marketdata.py

Why it's bad: np.allclose(a, b, rtol, atol) — the third positional argument is rtol (relative tolerance), not atol. Every call of the form np.allclose(test, ref, 1e-10) sets rtol=1e-10 while leaving atol at its default of 1e-8. For order-1 values this is invisible, but for tests on large-magnitude quantities (forwards, prices) or very small quantities (greeks, probability densities) the wrong axis is controlled and tests can pass for the wrong reasons. 

assert np.allclose(test, ref, 1e-10)                   # sets rtol, intent is atol
assert np.allclose(test, ref, rtol=0.0, atol=1e-10)    # explicit and correct
#10 Maximum and BarrierDown WIP stubs still in the public API, raise NotImplementedError at runtime Medium
sdevpy/montecarlo/payoffs/exotics.py · lines 71–96

Why it's bad: Both classes are tagged "work in progress", both raise NotImplementedError unconditionally in evaluate(), and both are importable from the public namespace with no underscore prefix or deprecation warning. The failure only surfaces deep inside the pricing engine at simulation time, not at construction. BarrierDown.__init__ also takes a raw integer asset_index rather than a name string, inconsistent with every other payoff class in the codebase.

Fix: Prefix as _Maximum, _BarrierDown to exclude them from the public API until complete, or add raise NotImplementedError(...) in __init__ itself so failure is immediate and clearly labelled at construction time.
