#2 New variance swaps always crash at pricing — current_fixing never set when start ≥ valdate Medium
sdevpy/montecarlo/payoffs/basic.py · lines 480–501, 463–464

Why it's bad: Variance.set_valuation_date() only assigns self.current_fixing when len(hist_fixings) ≥ 1 (line 500). For a brand-new contract where start ≥ valdate, all dates fall into self.eventdates, hist_fixing_dates = [], and get_fixings(name, []) returns nothing — the condition is never met. current_fixing stays None and evaluate() at line 463 raises ValueError on every pricing call. The existing test uses start = 2025-11-15 / valdate = 2025-12-15 (always has historical fixings) — the new-contract path is never exercised.

Fix: When hist_fixing_dates is empty, set current_fixing from market data (e.g. fxgs.get_fixing(name, valdate)), or expose a set_current_fixing(spot) method and document that it must be called before pricing a new contract.

#3 lv.section(exp_idx) does a time-lookup with an integer — semantically wrong, latent bug Medium
sdevpy/volatility/localvol/lvsection_calib.py · line 114

Why it's bad: section(t: float) finds the largest t_grid[i] ≤ t. When called as lv.section(exp_idx) with an integer like 0, 1, 2…, those integers are interpreted as time values in years. For exp_idx = 0 this accidentally returns sections[0]. For exp_idx = 1 it finds the last section with time ≤ 1.0 year — not necessarily sections[1]. The bug is currently silent because all BiExpSection instances return identical bounds from constraints(). If any future model uses per-section bounds, the calibration will silently apply wrong constraints at every expiry beyond the first. 

bounds = lv.section(exp_idx).constraints()
bounds = lv.section_at_index(exp_idx).constraints()
#4 Cashflow.notional and Trade.notional both applied — potential double-scaling Medium
sdevpy/montecarlo/payoffs/cashflows.py · line 13  ·  sdevpy/montecarlo/mcpricer.py · line 177

Why it's bad: Cashflow.calculate() returns payoff_amount * self.notional * self.accrual. Then MonteCarloPricer.build() multiplies the summed cashflows again by trade.notional. Any cashflow constructed with notional != 1.0 produces a price scaled by both factors. The fix that correctly applied trade.notional in the 9th review was right, but it exposed this pre-existing design ambiguity: both levels carry a notional with no documented contract for which one means what.

Fix: Clarify the contract in docstrings — e.g. Cashflow.notional is for day-count accrual fractions (default 1.0), Trade.notional is the face amount. Verify that all factory functions (make_vanilla_option, etc.) create cashflows with notional = 1.0 unless there is an explicit accrual to model.

#5 No tests for TsIvCalibrator.calibrate or calibrate_lv_bysections Medium
sdevpy/tests/ · codebase-wide

Why it's bad: The two main calibration entry points — TsIvCalibrator.calibrate() (impliedvol_calib.py) and calibrate_lv_bysections() (lvsection_calib.py) — are exercised only by notebooks and example scripts, never by pytest. Issue #3 above (wrong section index) and the dupire example bug (#1) would both be immediately visible if a round-trip calibration test existed. test_localvol_calib.py covers the builder internals but stops short of the full calibration loop.

Fix: Add (1) a test_tssvi1_calibrate that calls TsIvCalibrator.calibrate() against the existing ABC sample data and asserts RMSE < a generous threshold such as 10 bps; (2) a test_calibrate_lv_bysections_smoke that runs one expiry slice of LV calibration and checks the returned LV is non-trivial.
