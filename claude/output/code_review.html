#1 New variance swaps always crash at pricing — current_fixing is never set when start ≥ valdate Critical
sdevpy/montecarlo/payoffs/basic.py · lines 452, 462–466

Why it's bad: Variance.__init__ sets self.current_fixing = None. In set_valuation_date, when the trade has not yet started (start ≥ valdate), all dates go into self.eventdates and hist_fixing_dates is empty. get_fixings returns None or an empty list, so neither guard fires and current_fixing stays None. Then evaluate() unconditionally raises: 

if self.current_fixing is None:
    raise ValueError(f"Fixing not set for variance on {self.name}")

Every new-issue variance swap is unrunnable. The existing test uses start_date = 2025-11-15 with valdate = 2025-12-15, ensuring historical fixings are always present — the failing path (new contract) is never exercised by the test suite.

Fix: For a new contract the first simulated spot is the start fixing. Expose a set_current_fixing(spot) method and call it from the pricing engine before evaluate(), or document that the caller must supply it. An alternative is to have evaluate() treat current_fixing is None as meaning the first path value is the starting point: 

if self.current_fixing is None:
    # New contract: first simulated spot is the start fixing
    first_spot = spot_paths[:, 0]
    var_sum = var_sum + np.power(np.log(spot_paths[:, 1] / first_spot), 2)
    spot_paths = spot_paths[:, 1:]  # shift window
else:
    var_sum = var_sum + np.power(np.log(spot_paths[:, 0] / self.current_fixing), 2)
#2 Ruff rename introduced NameError in price_vanilla() — lambda body uses s but parameter is still S High
docs/stovol/products.py · lines 25, 31

Why it's bad: The ruff lowercase rename pass changed the body of the terminal lambdas from S - K / K - S to s - k / k - s, but left the lambda parameter as uppercase S. The lowercase s in the body refers to nothing in price_vanilla's local scope — it is an undefined name. Any call to price_vanilla() reaches the terminal condition and raises NameError: name 's' is not defined. 

# Current (broken) — S is the lambda param, s is undefined:
terminal = lambda S, v: np.maximum(s - k, 0.0) # noqa
terminal = lambda S, v: np.maximum(k - s, 0.0) # noqa

# Fix — lowercase the lambda parameter to match the body:
terminal = lambda s, v: np.maximum(s - k, 0.0)  # noqa: E731
terminal = lambda s, v: np.maximum(k - s, 0.0)  # noqa: E731

Note also that smax_bc at line 29 references solver.s[-1] — confirm the SLVPdeSolver attribute was also renamed from S to s in the same pass; if not, that will raise AttributeError.

#4 brownianbridge() outer factor loop is pure Python — factor dimension not vectorized Medium
sdevpy/maths/rand/pathconstruction.py · line 43

Why it's bad: The outer for d in range(n_factors) loop processes one factor at a time in Python, while all inner operations (mean, variance, midpoint assignment) are already fully vectorized over paths. With n_factors assets (e.g. a 5-asset worst-of basket), the loop runs 5 times, each time issuing separate NumPy calls. The incremental path builder has no such loop — both the path and factor dimensions are handled in a single vectorized block.

Fix: Broadcast over the factor dimension by keeping d as a full slice. All indexing into w and z already supports the factor axis; removing the outer loop and replacing [:, end, d] with [:, end, :] throughout should suffice: 

for d in range(n_factors):
    w[:, -1, d] = np.sqrt(t) * z[:, 0, d]
    ...
    w[:, mid, d] = mean + np.sqrt(var) * z[:, z_index, d]

w[:, -1, :] = np.sqrt(t) * z[:, 0, :]
...
mean = ((t_mid - t_start) * w[:, end, :] + (t_end - t_mid) * w[:, start, :]) / (t_end - t_start)
var  = (t_mid - t_start) * (t_end - t_mid) / (t_end - t_start)
w[:, mid, :] = mean + np.sqrt(var) * z[:, z_index, :]
#5 implied_vols() iterates over strikes with a Python loop — vectorized implied_vol_newton exists but is never called Medium
sdevpy/analytics/black.py · line 51

Why it's bad: implied_vols() runs a serial Python list comprehension calling the scalar Brent solver once per strike. For a 50-strike surface this is 50 sequential Python → SciPy calls. implied_vol_newton() in the same file is a fully vectorized Newton-Raphson solver that processes the whole strike array in one call, typically converging in under 10 iterations. It is defined, but implied_vols() never delegates to it. 

# Current (slow — one Brent call per strike):
ivs = [implied_vol(expiry, k, is_call, fwd, p) for k, p in zip(strike, fwd_price, strict=True)]

# Drop-in replacement (vectorized Newton, ~10–100× faster):
return implied_vol_newton(expiry, strike, is_call, fwd, fwd_price)

The implied_vol_newton docstring notes it may struggle near zero vega; a safe drop-in would add a Brent fallback for those cases. As a first step, delegating unconditionally is a large improvement with minimal risk.

#7 Arithmetic payoff nodes repeat identical 4-method boilerplate across 4 classes (~130 lines) Low
sdevpy/montecarlo/payoffs/basic.py · lines 514–619

Why it's bad: Add, Sub, Mul, and Div each implement the same four lifecycle methods (set_nameindexes, set_valuation_date, set_eventindexes, and __init__) with identical structure — only the operator in evaluate() differs. A bug in one copy (e.g. a missing delegation to right.set_eventindexes) would be silently inconsistent across the other three.

Fix: Extract a BinaryPayoff base class that handles all lifecycle delegation and exposes an abstract _combine(left_val, right_val) method: 

class BinaryPayoff(Payoff):
    def __init__(self, left, right):
        super().__init__()
        self.left, self.right = left, right
        self.names = list_payoff_names([left, right])

    def set_nameindexes(self, names): ...   # delegate to both
    def set_valuation_date(self, valdate): ...  # delegate + gather eventdates
    def set_eventindexes(self, eventdates): ...  # delegate to both

class Add(BinaryPayoff):
    def evaluate(self, mkt_state):
        return self.left.evaluate(mkt_state) + self.right.evaluate(mkt_state)

#10 implied_vol_newton() has no vega guard — division by near-zero vega can diverge silently Low
sdevpy/analytics/black.py · lines 73–76

Why it's bad: The Newton loop computes vol -= diff / vega where vega = fwd * norm.pdf(d1) * sqrt_t. For very deep in-the-money or out-of-the-money options, norm.pdf(d1) underflows to zero and the Newton step overflows to ±inf. The clamp np.maximum(vol, 1e-8) on the next line saves negative vols but not +inf, which propagates silently into downstream pricing. The returned array can contain inf with no warning. 

vol -= diff / vega
step = np.where(vega > 1e-12, diff / vega, 0.0)  # skip near-zero-vega points
vol -= step

Points skipped in a given iteration are re-evaluated on the next iteration; in practice they converge once the vol estimate moves to a more sensible region. Alternatively, add a post-loop sanity check: vol = np.where(np.isfinite(vol), vol, np.nan) and raise if any NaNs remain.
