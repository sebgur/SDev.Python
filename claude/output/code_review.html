#1 Missing assert in test_calib_dupire() — LV values never checked High New
sdevpy/tests/test_localvol.py : 161

The last line of test_calib_dupire() calls np.allclose(lv_test, lv_ref, 1e-10) but forgets the assert. The return value is silently discarded, so the test passes regardless of whether the computed local volatilities match the reference values. This is the most important regression in this batch: the entire point of the test is to pin down the numerical output of calib_lv_dupire(), and that check is not running. 

-    np.allclose(lv_test, lv_ref, 1e-10)
+    assert np.allclose(lv_test, lv_ref, 1e-10)
#2 Debug print() re-introduced in dupire_formula_single() High New
sdevpy/volatility/localvol/dupire.py : 32

print(f'dvar_dt: {dvar_dt}') was added inside the scalar Dupire function, which was already flagged for deletion. This is the same class of raw-print regression that was cleaned up two reviews ago from density(). Any caller or test that exercises this path will produce stdout noise — including pytest output unless -s is passed.

Fix: Delete the print() call. If the value needs observability, replace with log.debug("dvar_dt: %s", dvar_dt) inside a if log.isEnabledFor(logging.DEBUG): guard so it is inert in production.

#3 Module-level log.setLevel(logging.DEBUG) forces DEBUG globally High New
sdevpy/volatility/localvol/dupire.py : 13  ·  sdevpy/utilities/timegrids.py : 10

Both modules call log.setLevel(logging.DEBUG) at import time. This overrides whatever log level the application or test runner configured and forces DEBUG output for anyone who imports these modules — even in production. Logging levels are a deployment concern, not a module concern. The logger itself should be created at its default level; users enable DEBUG via logging.basicConfig(level=logging.DEBUG) or their own configuration. 

-log.setLevel(logging.DEBUG)
# Remove this line entirely — callers control the level

The same pattern exists in other modules in the codebase (localvol_calib.py, localvol.py, etc.); it should be removed from all of them.

#4 Infeasibility penalty is scale-dependent across expiries High
sdevpy/volatility/localvol/localvol_calib.py : 189

When LvObjectiveBuilder.check_params() returns infeasible, the objective returns self.cf_prices.sum(). This sum depends on the number of strikes and their absolute price levels at the current expiry. As the outer calibration loop advances through expiries, the penalty magnitude shifts — so the optimizer sees an inconsistent cost landscape: equally infeasible parameters produce different penalty values at different expiries. The comment directly above notes the open question about FLOAT_INFTY; TsIvObjectiveBuilder already uses it consistently. 

-            return self.cf_prices.sum()
+            return constants.FLOAT_INFTY

#6 Wrong return-type annotation on get_grid() Medium
sdevpy/utilities/timegrids.py : 73

get_grid() is annotated -> npt.NDArray(np.float64) with parentheses — this is a function call expression, not a type annotation. The correct subscript syntax is npt.NDArray[np.float64] with square brackets. The parentheses form evaluates to an ndarray instance at import time, making the annotation a runtime object rather than a type. Static type-checkers (mypy, pyright) will reject it. 

-    def get_grid(self) -> npt.NDArray(np.float64):
+    def get_grid(self) -> npt.NDArray[np.float64]:
#7 Duplicate validation in TsSvi1/TsSvi2.smile_parameters() Medium
tssvi1.py : 57–61 vs 112–118  ·  tssvi2.py : 53–57 vs 100–106

check_global_params() validates r and delta (TsSvi1) / rho0 and rhoinf (TsSvi2) and returns (False, FLOAT_INFTY) on failure. calculate() calls check_global_params() first, so smile_parameters() is never reached with invalid correlation through the normal calibration path. Despite this, smile_parameters() re-validates the same guards and raises ValueError. The same invalid input raises an exception from one entry point and returns a penalty tuple from another — an inconsistent contract that makes defensive callers impossible.

Fix: Remove the ValueError guards from smile_parameters() in both models; check_global_params() is the single validation point.

#8 make_schedule() contains unverified production code path Medium
sdevpy/utilities/scalendar.py : 71–91

The comment on line 72 reads "New logic (to be tested)" while the old implementation is preserved as commented-out code at lines 85–89 labelled "Old logic". This signals that the new schedule generation code is unverified and the old fallback has been kept for rollback. Shipping code annotated "to be tested" is a reliability risk; if the logic is correct it should be tested and the comment removed; if it is not yet correct it should not be in the default branch.

Fix: Write a test for make_schedule() (distinct from the existing test_make_schedule() which calls make_schedule_fancy() directly), verify it passes, then remove the "New logic (to be tested)" comment and the commented-out old implementation.

#9 svi() returns a NaN array on invalid parameters Medium
sdevpy/volatility/impliedvol/models/svi.py : 40–42

When svi_check_params() reports infeasible, svi() returns np.full_like(x, np.nan). This is inconsistent with every other model in the codebase — svi_check_params() itself returns FLOAT_INFTY as a penalty. Any downstream RMSE calculation receiving NaN will silently propagate NaN through the optimizer without triggering a recognisable error signal. The commented-out alternative (# raise ValueError("Invalid SVI parameters")) shows this was a deliberate trade-off that was never evaluated. 

-        return np.full_like(np.asarray(x, dtype=float), np.nan)
+        raise ValueError("Invalid SVI parameters")
#10 dupire_formula_single() is pending-removal dead code with a new debug print Low
