#1 Maximum and BarrierDown are incomplete WIP stubs exported in the public API High
sdevpy/montecarlo/payoffs/exotics.py · lines 71–96

Why it's bad: Both classes carry "Warning: work in progress" comments and both raise NotImplementedError in evaluate(). They are importable from the public namespace. Maximum.__init__ also sets self.eventdates at construction time — the wrong lifecycle phase — rather than in set_valuation_date, inconsistent with every other payoff in the codebase. Any caller who accidentally instantiates either class gets a runtime error deep inside the pricing engine with no indication at the call site.

Fix: Until fully implemented, rename to _Maximum / _BarrierDown (underscore-prefix = not public API), or add raise NotImplementedError in __init__ so the failure is immediate and clearly labelled at construction time, not at pricing time.

#2 test_localvol_calib.py duplicates 8 tests verbatim from test_localvol.py, one regressing to rounded references Medium
sdevpy/tests/test_localvol_calib.py · lines 23–164  ·  sdevpy/tests/test_localvol.py · lines 155–433

#3 Per-strike price loops in forwardpde.py and lvsection_calib.py are the remaining hot-path Python loops Medium
sdevpy/pde/forwardpde.py · line 196  ·  sdevpy/volatility/localvol/lvsection_calib.py · line 191

Why it's bad: Both sites iterate over strikes in pure Python and accumulate np.trapezoid calls one-by-one — the same pattern just vectorized in the PDE scheme matrices. In lvsection_calib.py, this inner loop runs inside the optimizer's objective function, which is called thousands of times per calibration. Both sites carry existing "ToDo: can this be vectorized?" comments. 

# lvsection_calib.py:191 — current
for k in self.strikes:
    payoff = np.maximum(s - k, 0.0)
    pde_prices.append(np.trapezoid(payoff * p, x))

# Vectorized replacement
k_2d = np.asarray(self.strikes)[:, None]   # (n_strikes, 1)
payoffs = np.maximum(s[None, :] - k_2d, 0.0)  # (n_strikes, n_x)
pde_prices = np.trapezoid(payoffs * p[None, :], x)  # (n_strikes,)

The same broadcast pattern removes the for k loop in forwardpde.py, where straddles can be expressed as np.abs(s[None,:] - k_2d).

#4 build_spotgrid fills a NumPy array with a Python for loop Medium
sdevpy/pde/forwardpde.py · lines 74–81

Why it's bad: The spot grid is a simple symmetric linspace around zero. The current implementation allocates x_grid with np.zeros and fills it element-by-element with a Python loop. This function is called at the start of every density step when rescale_x=True — once per time section per calibration iteration. 

# Current (lines 74–81)
x_grid = np.zeros(2 * n_half + 1)
x = 0.0
for i in range(n_half):
    x = x + dx
    x_grid[n_half + 1 + i] = x
    x_grid[n_half - 1 - i] = -x

# Vectorized replacement — one line, identical result
x_grid = np.arange(-n_half, n_half + 1) * dx

#6 New variance swap (start ≥ valdate) crashes at pricing — current_fixing never set Medium
sdevpy/montecarlo/payoffs/basic.py · lines 452, 497–501

Why it's bad: Variance.__init__ sets self.current_fixing = None. In set_valuation_date, when start ≥ valdate (a brand-new swap), hist_fixing_dates = [], so get_fixings returns nothing, the guard if hist_fixings is not None and len(hist_fixings) >= 1 is never met, and current_fixing stays None. Then evaluate() raises: 

if self.current_fixing is None:
    raise ValueError(f"Fixing not set for variance on {self.name}")

The existing test always uses start = 2025-11-15, valdate = 2025-12-15 (historical fixings exist) — the failing case is never covered.

Fix: When hist_fixing_dates is empty, seed current_fixing from the current spot (via market data), or expose a set_current_fixing(spot) method and document it as required before pricing a brand-new swap.

#7 PdeConfig.shift_forward defaults to True but the function docstring says "not using this for now" Low
sdevpy/pde/pdeschemes.py · line 20  ·  sdevpy/pde/forwardpde.py · lines 102–103, 171

Why it's bad: PdeConfig declares shift_forward: bool = True, meaning any caller who constructs a bare PdeConfig() silently runs the forward-shift correction. But shift_forward() carries the docstring "Shift the density to match the forward. We are not using this for now." And price_vanillas (the only production caller) explicitly overrides to shift_forward=False. A contributor adding a new caller with default config would silently exercise code the author labelled as unused.

Fix: Change the default to False to match actual production usage, and rewrite the docstring to describe what the function does (not its usage status).

#8 LvObjectiveBuilder infeasibility penalty scales with expiry level, making it inconsistent across sections Low
sdevpy/volatility/localvol/lvsection_calib.py · line 211

Why it's bad: When parameters fail the validity check, the objective returns self.cf_prices.sum() — the total call option price across all strikes at the current expiry. Longer-dated options are worth more (higher forward prices), so the penalty grows with expiry. The optimizer at section 5 (2y) faces a much larger "infeasible" signal than at section 1 (1m), making step-size control and tolerance comparisons inconsistent across sections. The in-code comment explicitly acknowledges this uncertainty.

Fix: Use a fixed, large constant penalty independent of price level — e.g. constants.FLOAT_INFTY or a hard-coded value like 1.0 (prices are normalised forward prices, so this is already large). The old concern about FLOAT_INFTY causing optimizer issues should be tested directly rather than replaced with a moving penalty.

#9 TsIvObjectiveBuilder silently accepts mkt_prices=None, crashing only at optimization time for ForwardPremium models Low
sdevpy/volatility/impliedvol/impliedvol_calib.py · lines 14–34

Why it's bad: In __init__, when calculate_type is ForwardPremium, self.target_values = self.market_prices. Test code passes mkt_prices=None for LogNormalVol models (which take the vol branch), so the bug is latent. If a ForwardPremium model is ever passed with mkt_prices=None, the objective silently stores None and crashes with a confusing TypeError inside rmse() deep in the optimizer, not at the constructor call.

if self.model.calculate_type == OptionQuoteType.ForwardPremium and mkt_prices is None:
    raise ValueError("mkt_prices is required when calculate_type is ForwardPremium")
#10 ImpliedVol.taylor_dx uses hardcoded hr = 0.05 with no documentation of known accuracy limits Low
sdevpy/volatility/impliedvol/impliedvol.py · lines 62–73

Why it's bad: The finite-difference bump size hr = 0.05 (5% relative) is a magic number shared across the Dupire formula, the probability density, and the calibration objective. The code shows two commented-out alternatives (0.01 and 0.10), implying empirical tuning. At very deep OTM moneyness (x ≈ 0), the bump 0.05 * x becomes negligible and the finite-difference quotient becomes numerically noisy, silently contaminating Dupire local vol and density calculations. No test verifies gradient accuracy as a function of moneyness or expiry.

Fix: Document why 0.05 was chosen and its known accuracy limits. Consider blending absolute and relative sizing: dx = max(hr * abs(x), x_min_abs) with a floor like 1e-4. Models with closed-form derivatives (SVI has analytic dσ/dx) should use them instead of finite differences.
