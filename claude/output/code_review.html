#1 Debug print() in density_step floods stdout on every PDE step when rescale_p is enabled High
sdevpy/pde/forwardpde.py · line 33

Why it's bad: Inside density_step, the mass-rescaling branch runs at the end of every density step and unconditionally prints the mass: 

if config.rescale_p:
    mass = np.trapezoid(p, x)
    print(f"Mass: {mass:.6f}")
    p /= mass

density_step is called once per maturity in calculate_densities, and calculate_densities is itself called inside the LV calibration objective (LvObjectiveBuilder.objective) that the optimizer evaluates many times per iteration. With rescale_p=True being the default in PdeConfig and used throughout the test suite, this floods stdout with thousands of "Mass: ..." lines during any non-trivial calibration, breaks output-parsing pipelines, and meaningfully slows the calibration through console I/O.

Fix: remove the line, or replace with logging.debug():

import logging
log = logging.getLogger(__name__)
...
log.debug("Mass: %.6f", mass)
#2 Debug print() in TimeInterpolatedLocalVol.section() floods stdout on every section retrieval High
sdevpy/volatility/localvol/localvol.py · line 86

Why it's bad: section() is called on every PDE roll (pdeschemes.ThetaScheme.roll_forward calls self.local_vol(ts, x) and self.local_vol(te, x), which reach TimeInterpolatedLocalVol.value → section), and on every MC step inside MultiAssetGBM.evolve_state. For a typical setup of 50 time steps × hundreds of optimizer evaluations, this prints tens of thousands of lines per calibration: 

def section(self, t: float) -> LocalVolSection:
    t_idx = algos.lower_bound(self.t_grid, t, clamp=True)
    print(f"Section requested at {t}, using pillar at time index/time: {t_idx}/{self.t_grid[t_idx]}")
    return self.section_at_index(t_idx)

The cost is identical in nature to issue #1 but worse in volume because section retrieval sits in the inner numerical loops, not the outer per-maturity loop.

Fix: remove the line, or downgrade to logging.debug() guarded on a verbosity flag.

#3 Trade.notional is set in the constructor but never read anywhere — all trades priced at unit notional High
sdevpy/montecarlo/payoffs/basic.py · lines 95–99  ·  sdevpy/montecarlo/mcpricer.py · lines 144–174

Why it's bad: Trade.__init__ stores self.notional = kwargs.get('notional', 1.0), but a project-wide search for trade.notional / t.notional returns zero hits. MonteCarloPricer.build iterates for trade in book.trades and aggregates only the cashflow PVs (cf.calculate(mkt_state) uses the cashflow notional, not the trade notional). Any caller that constructs Trade(instrument, notional=10.0) gets the same PV as Trade(instrument) — silently. This is a real correctness bug for any portfolio where trade-level notionals are used.

Fix: either drop the parameter or apply it in build():

results['pv'] = trade.notional * np.asarray(leg_cf_pvs).sum(axis=1).sum()

Ideally, also clarify with a test (e.g. test_trade_notional_scales_pv) that doubling notional doubles PV.

#4 Flat-vol sigma still hard-wired at TEST_VOL = 0.20 — not injectable High
sdevpy/models/assetmodels.py · lines 5, 25

Why it's bad: When use_lv=False (the default), every asset is simulated at exactly 20% flat volatility regardless of market data. The constant name TEST_VOL signals the value is for tests only, but no kwargs hook exists to override it. Any production-style use of MultiAssetGBM with the constant-vol path silently yields wrong prices. Flagged in the prior two reviews. 

class MultiAssetGBM(FactorModel):
    def __init__(self, spot, fwd_curve, lv, time_grid, **kwargs):
        ...
        self.sigma = np.asarray([TEST_VOL for _ in range(self.n_factors)])
        self.use_lv = kwargs.get('use_lv', False)

Fix:

sigma = kwargs.get('sigma', [TEST_VOL] * self.n_factors)
self.sigma = np.asarray(sigma)
if self.sigma.shape[0] != self.n_factors:
    raise ValueError(f"sigma has {self.sigma.shape[0]} entries, expected {self.n_factors}")
#5 Inconsistent exception types for the same precondition: RuntimeError vs ValueError for "Past trade found" Medium
sdevpy/montecarlo/payoffs/basic.py · lines 218–220, 421–423

Why it's bad: Terminal.set_valuation_date raises RuntimeError("Past trade found") while WorstOf.set_valuation_date raises ValueError("Past trade found") for the exact same condition. Any caller that wants to handle "expired trade" centrally must catch both, and one is conventionally wrong: Python's convention is that ValueError signals an input value that is structurally valid but semantically wrong, which fits here. 

# Terminal:
raise RuntimeError("Past trade found")

# WorstOf:
raise ValueError("Past trade found")

Fix: standardise on ValueError across both payoffs (and any future ones); the codebase has many other RuntimeError uses for value-domain checks (eqforward.py:76, cubicvol.py:140–159, logmix.py:196–284) that should be migrated for the same reason.

#6 Dead code: TsIvCalibrator.check_consistency is never called and its conversion path is bypassed Medium
sdevpy/volatility/impliedvol/impliedvol_calib.py · lines 129–143

Why it's bad: The method has a self-aware comment — "this function is not used yet, but will be in a later phase if/when we want to introduce negative rates" — yet it is the only caller of convert_to_target_values and keep_positive. The bugfix referenced by the prior review (skip empty expiry sublists in convert_to_target_values) is therefore dead-code-protected: TsIvCalibrator.calibrate never invokes it. Either the conversion logic should be wired up (and the empty-sublist fix is genuinely load-bearing), or the whole branch should be deleted to avoid the false sense of safety.

Fix: decide. If the conversion path is the intended future, delete the "not used yet" comment and call self.check_consistency(option_targets) from prepare_target_data. Otherwise delete check_consistency, convert_target, convert_option, and convert_to_target_values until they are needed.

#7 dump_data calls self.params.tolist() without ensuring params is a NumPy array Medium
sdevpy/volatility/impliedvol/models/tssvi1.py · line 157  ·  tssvi2.py · line 148  ·  logmix.py · line 366

Why it's bad: The base-class type hint says update_params(self, x: list[float]) and self.params = x performs no conversion. If a caller passes a Python list (which the docstring explicitly invites), dump_data raises AttributeError: 'list' object has no attribute 'tolist'. The current tests happen to pass np.asarray(...) from initial_point(), but several call sites in the same files use Python lists (FLAT_TSSVI1_PARAMS in test_localvol.py:25, the params list in test_tssvi1).

Fix: coerce to ndarray on assignment, or convert defensively in dump_data:

def update_params(self, x: list[float]) -> None:
    self.params = np.asarray(x, dtype=float)
#8 Maximum and BarrierDown are still importable WIP stubs in the public payoffs API Medium
sdevpy/montecarlo/payoffs/exotics.py · lines 71–96

Why it's bad: Both classes raise NotImplementedError in evaluate (improvement over the silent no-op of two reviews ago) but they are still importable from sdevpy.montecarlo.payoffs.exotics with no early guard and no name-resolution lifecycle. Maximum.__init__ sets self.eventdates at construction time (wrong phase — should happen in set_valuation_date), and BarrierDown.__init__ takes a raw integer asset_index bypassing set_nameindexes entirely. Anyone who imports them, instantiates them, and adds them to a book gets a NotImplementedError only at evaluate time — far from the construction site.

Fix: raise immediately in __init__ with a clear message, or move both classes to a private _wip.py module.

class Maximum(Payoff):
    def __init__(self, left, right):
        raise NotImplementedError("Maximum payoff is WIP — do not use yet")
#9 yieldcurve_from_file stores snapdate as a raw string instead of a datetime Medium
sdevpy/market/yieldcurve.py · lines 163–172

Why it's bad: data.get('snapdate') returns the JSON string, and the call to InterpolatedYieldCurve(..., snapdate=snapdate) stores it verbatim. Any later call to curve.snapdate.strftime(...) (e.g. from dump_data at line 133) will then fail with AttributeError. The inline comment "Claude says this will be wrong" already flags the bug — it should be fixed, not annotated. The reason the round-trip currently works in unit tests is that none of them re-dump the loaded curve. 

name = data.get('name')
valdate = data.get('valdate')
snapdate = data.get('snapdate') # Claude says this will be wrong
# Claude says to do the below but this fails in UT
# snapdate = dt.datetime.strptime(data.get('snapdate'), dts.DATE_FILE_FORMAT)

Fix: the format mismatch is real — the file stores DATETIME_FORMAT (per dump_data), not DATE_FILE_FORMAT: 

snapdate = dt.datetime.strptime(data.get('snapdate'), dts.DATETIME_FORMAT)

and add a round-trip test that dumps, reloads, and dumps again.

#10 No tests for TsIvCalibrator.calibrate, calibrate_lv_interp, or the shift_forward-enabled PDE path Medium
sdevpy/tests/ · codebase-wide

Why it's bad: The new test files cover smile-level evaluation (test_impliedvol.py) and the per-section LV objective (test_localvol.py::test_lv_bysections_builder_*), but the calibration entry points themselves — TsIvCalibrator.calibrate, calibrate_lv_interp, calib_lv_dupire's full pipeline at production sizes — are exercised only via __main__ blocks. The density_step branch with config.shift_forward = True is also unverified, even though the previous review's critical bug lived in exactly that branch. The bugs surfaced in this review (issues #1, #3, #6, #9) would all be caught by basic end-to-end smoke tests.

Fix: add at minimum (a) a test_tsivcalibrator_smoke that runs TsSvi1 calibration on a small fixture and asserts RMSE < some tolerance; (b) a test_density_step_with_shift_forward that toggles shift_forward=True and asserts the returned density integrates to 1.0; (c) a test_trade_notional that doubles a notional and checks PV scales.
