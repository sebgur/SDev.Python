#1 Filename format mismatch between write_example() and load_lv_from_folder() High New
sdevpy/volatility/localvol/localvol_factory.py : 73, 153

The two functions are intended to be inverses (write a file, load it back), but they construct the filename differently: 

# write_example (line 153)
file = os.path.join(file, "localvol_" + date.strftime("%Y%m%d-%H.%M.%S") + ".json")

# load_lv_from_folder (line 73) — uses dates.DATE_FILE_FORMAT = "%Y%m%d-%H%M%S"
file = os.path.join(file, store_date.strftime(dates.DATE_FILE_FORMAT) + ".json")

A datetime 2025-12-15 12:30:45 serialises to localvol_20251215-12.30.45.json on write but the loader looks for 20251215-123045.json. Anything written by write_example cannot be read by load_lv_from_folder, and the failure is a silent FileNotFoundError at load time rather than a clear API error.

Fix: Use dates.DATE_FILE_FORMAT in both places, and decide once whether the file should have the localvol_ prefix:

-    file = os.path.join(file, "localvol_" + date.strftime("%Y%m%d-%H.%M.%S") + ".json")
+    file = os.path.join(file, date.strftime(dates.DATE_FILE_FORMAT) + ".json")

A round-trip test (write_example() then load_lv_from_folder() on the same date) would catch this in CI.

#2 Open-questions TODO block at the top of localvol_factory.py Medium New
sdevpy/volatility/localvol/localvol_factory.py : 10–18

The module begins with a six-line TODO block consisting entirely of open design questions: 

############ TODO #######################################################
# * Plug in matrix local vols. But how do we do that? Can we also implement
#   a calibration on the fly? We need to plug the two sides of the design.
# * Maybe we can do that by an intermediate step where we dump the matrices
#   in text files?
# * So maybe here we need to implement that reading and start unifying the
#   different local vol types between InterpolateSection and Matrix?
# * Maybe we try to find a file and use it if it's there, and if it's not
#   there then we trigger the calibration? And then the calibration dumps?

Unanswered design questions in production source code create ambiguity about whether the module is ready to be consumed. The factory's API contract is the central concern of any consumer, and reading "we need to plug the two sides of the design" tells them the contract is not stable.

Fix: Move open design questions to a design doc, an issue tracker, or a planning notebook. Keep the source file declarative about what it currently does.

#10 implied_vols() return type is inconsistent in the scalar branch Low
sdevpy/analytics/black.py : 41–42

implied_vols() is annotated -> npt.NDArray[np.float64], but when both strike and fwd_price are scalar it returns the bare float from implied_vol(). Callers that unconditionally call array methods (.shape, indexing) on the result will fail at runtime for scalar inputs without a clear type error. 

-        return implied_vol(expiry, strike, is_call, fwd, fwd_price)
+        return np.float64(implied_vol(expiry, strike, is_call, fwd, fwd_price))

Alternatively, update the annotation to -> float | npt.NDArray[np.float64] if the scalar path is intentional.
