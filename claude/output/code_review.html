#1 LvObjectiveBuilder.calculate_vols() compares a plain Enum to integer literals — call and put branches are permanently unreachable High
sdevpy/volatility/localvol/lvsection_calib.py · lines 329–335

Why it's bad: self.option_type is set by string_to_optiontype(option_type) which returns a plain OptionType enum member. OptionType inherits from Enum, not IntEnum. In Python, a plain Enum member is never equal to its underlying integer value: OptionType.CALL == 0 evaluates to False. The guard conditions are: 

if self.option_type == 0:   # Always False — OptionType.CALL != 0 for plain Enum
    pde_vols.append(black.implied_vol(expiry, k, True, self.fwd, p))
elif self.option_type == 1:  # Always False — OptionType.PUT != 1 for plain Enum
    pde_vols.append(black.implied_vol(expiry, k, False, self.fwd, p))
else:                         # Always reached — straddle branch fires for ALL option types
    call = (p - k + self.fwd) / 2.0
    pde_vols.append(black.implied_vol(expiry, k, True, self.fwd, call))

When option_type='call' or 'put', the code silently executes the straddle-to-call conversion on a non-straddle price, producing numerically wrong implied vols with no error or warning. The rest of the codebase (e.g. forwardpde.py:187) uses the correct case OptionType.CALL: match-statement syntax.

Fix: Replace integer comparisons with enum members:

match self.option_type:
    case OptionType.CALL:
        pde_vols.append(black.implied_vol(expiry, k, True, self.fwd, p))
    case OptionType.PUT:
        pde_vols.append(black.implied_vol(expiry, k, False, self.fwd, p))
    case OptionType.STRADDLE:
        call = (p - k + self.fwd) / 2.0
        pde_vols.append(black.implied_vol(expiry, k, True, self.fwd, call))
#2 InterpolatedYieldCurve.set_data() uses unsorted dfs in all three interpolation branches — mismatched grid when pillars passed out of order High
sdevpy/market/yieldcurve.py · lines 79, 85

Why it's bad: set_data() sorts pillars by date into self.dates and self.dfs. The interpolation grid is then built using times derived from the sorted self.dates, but both the ZERORATE and DISCOUNT/LOG_DISCOUNT branches use the original unsorted parameter dfs: 

# ZERORATE branch (line 79)
times = timegrids.model_time(self.valdate, self.dates)  # sorted ✓
data_y = -np.log(dfs) / times                        # unsorted parameter ✗

# DISCOUNT / LOG_DISCOUNT branch (line 85)
ext_dfs = [1.0]
ext_dfs.extend(dfs)  # unsorted parameter ✗

When pillar dates arrive in non-chronological order, times is sorted but data_y retains the original ordering, producing a mismatched interpolation grid that returns wrong discount factors silently. The existing test only passes sorted dates, so the bug is invisible in CI. The fix is one word in each branch: replace dfs with self.dfs.

Fix:

# ZERORATE
data_y = -np.log(dfs) / times
data_y = -np.log(self.dfs) / times

# DISCOUNT / LOG_DISCOUNT
ext_dfs = [1.0]
ext_dfs.extend(dfs)
ext_dfs.extend(self.dfs)
#3 Variance payoff always raises at pricing when contract start ≥ valdate High
sdevpy/montecarlo/payoffs/basic.py · lines 452, 496–501, 463–464

Why it's bad: Variance.__init__ sets self.current_fixing = None. In set_valuation_date(), when start ≥ valdate (a brand-new variance swap), all dates go into self.eventdates and hist_fixing_dates is empty. The guard at line 500 — if hist_fixings is not None and len(hist_fixings) ≥ 1 — is never satisfied, so current_fixing stays None. Then evaluate() at line 463 unconditionally raises: 

if self.current_fixing is None:
    raise ValueError(f"Fixing not set for variance on {self.name}")

The only test in test_mc.py uses start_date = 2025-11-15 (one month before valdate = 2025-12-15), so historical fixings always exist and the failing new-contract case is never exercised.

Fix: When hist_fixing_dates is empty, initialise current_fixing from the current spot in market data, or expose a set_current_fixing(spot) method and document it as a pre-pricing requirement.

#4 Hard-coded name_model_map in localvol_factory.py silently breaks for any real ticker Medium
sdevpy/volatility/localvol/localvol_factory.py · line 19

Why it's bad: The factory contains a module-level dict: 

name_model_map = {'ABC': 'BiExp', 'KLM': 'VSVI', 'XYZ': 'Matrix'}

Any call to load_param_lv(date, name) or get_local_vols(names, valdate) without an explicit model_name keyword falls back to this map. For any ticker outside these three test fixtures, model_name becomes None and the function raises ValueError: No model name specified for name: …. Since get_local_vols is called from price_book in the MC pricer, any production deployment pricing real instruments fails immediately.

Fix: Remove the hard-coded map and require callers to always supply model_name explicitly, or drive the mapping from a runtime configuration file.

#5 Maximum and BarrierDown are WIP stubs exported publicly with no early failure in __init__ Medium
sdevpy/montecarlo/payoffs/exotics.py · lines 71–96

Why it's bad: Both classes are tagged "ToDo: [Warning] work in progress" and both raise NotImplementedError only in evaluate(). They are importable from the public sdevpy.montecarlo.payoffs.exotics namespace. Maximum.__init__ sets self.eventdates at construction time (wrong lifecycle — should be deferred to set_valuation_date). BarrierDown stores a raw integer index instead of a name, bypassing the name-resolution system entirely. A caller who accidentally instantiates either class gets a runtime error deep inside the pricing engine with no indication of why.

Fix: Until fully implemented, prefix with an underscore (_Maximum, _BarrierDown) to remove from the public API, or add raise NotImplementedError in __init__ itself for immediate and clearly labelled failure.

#6 black.implied_vol caps at xmax=2.0 — silently wrong for vols above 200% Medium
sdevpy/analytics/black.py · lines 35–48

Why it's bad: implied_vol minimizes squared error on the interval [1e-6, 2.0]. Any option whose true implied vol exceeds 200% — short-dated deep OTM options, high-vol names, or variance swap component inversions — returns the clipped value 2.0 without raising an error. This function is called from LvObjectiveBuilder.calculate_vols() and optionsurface.convert_option(), so calibration quality metrics can be silently wrong. A vectorized Newton-Raphson implementation (implied_vol_newton) already exists in the same file and handles this case correctly. 

xmax = 2.0
xmax = 10.0  # or replace call with implied_vol_newton
#7 matplotlib.pyplot unconditionally imported at module level in two pure-math files Medium
sdevpy/maths/optimization.py · line 10  ·  sdevpy/maths/interpolation.py · line 5

Why it's bad: Both files import import matplotlib.pyplot as plt unconditionally at the top of the module, even though plt is only used inside if __name__ == "__main__": blocks at the very bottom of each file. As a result, any code that imports sdevpy.maths.optimization or sdevpy.maths.interpolation — which is essentially the entire library — immediately triggers a full matplotlib import. This imposes a hard runtime dependency on a heavy plotting library for purely numeric code, increases import time noticeably in headless/server environments, and can cause errors in environments where a display is not available (e.g. CI, Docker without MPLBACKEND set).

Fix: Move the import inside the if __name__ == "__main__": guard in both files: 

if __name__ == "__main__":
    import matplotlib.pyplot as plt
    ...
#8 Variance payoff has an off-by-one in its n_returns calculation when historical fixings are present Medium
sdevpy/montecarlo/payoffs/basic.py · lines 453–454, 466–471

Why it's bad: Variance.__init__ computes: 

self.n_dates = len(self.alldates)
self.n_returns = len(self.alldates) - 1

This sets n_returns to the total number of returns from start to end using all fixing dates. However, when the trade is already live (start < valdate), some fixings become historical and some remain forward-looking. In evaluate(), the normalization denominator self.n_returns remains the full count. But the actual number of returns summed in var_sum is: len(hist_fixings) - 1 (historical log-returns) plus 1 (current increment bridging historical to live) plus len(self.varidxs) (forward log-returns from paths). These components add up correctly to n_returns only when the schedule is continuous — any gap (e.g. skipped business days due to the calendar) silently miscounts, biasing the realized variance estimate.

Fix: Track n_hist_returns in set_valuation_date() and compute n_fwd_returns = len(self.eventdates) - 1 (or use len(self.varidxs) - 1), then assert n_hist_returns + n_fwd_returns + 1 == self.n_returns.

#9 No test for InterpolatedYieldCurve.set_data() with unsorted pillar dates Medium
sdevpy/tests/test_yieldcurves.py · codebase-wide

Why it's bad: The only two yield curve tests (test_yieldcurve_creation and test_yieldcurve_reading) both pass pillar dates in strictly chronological order. The sort logic in set_data() is therefore never exercised. The unsorted-dfs bug described in issue #2 would produce wrong discount factors for any out-of-order input, but it cannot be caught by the current test suite. Real trading systems regularly update curves from unsorted data feeds.

Fix: Add a test that passes pillars in reverse chronological order and asserts that the resulting discount factors match those obtained from the same pillars in sorted order: 

def test_yieldcurve_sorted_regardless_of_input_order():
    # Pass same pillars in reverse order; result must be identical
    curve_fwd = make_curve(zdates, dfs)
    curve_rev = make_curve(zdates[::-1], dfs[::-1])
    assert np.allclose(curve_fwd.discount(test_dates), curve_rev.discount(test_dates))
#10 LocalVol.__init__ defaults valdate to dt.datetime.now() — mutable default evaluated at import time Low
sdevpy/volatility/localvol/localvol.py · line 38

Why it's bad: The base class LocalVol.__init__ defaults valdate to dt.datetime.now(): 

def __init__(self, **kwargs):
    self.valdate = kwargs.get('valdate', dt.datetime.now())

dt.datetime.now() is called at object construction time (inside kwargs.get), not at class definition time, so there is no single frozen timestamp shared across all instances. However, the practical consequence is that any LocalVol subclass instantiated without an explicit valdate silently gets the current wall-clock time as its valuation date. This makes dump_data() produce timestamps that differ between runs, which can cause snapshot-test failures and makes reproducibility debugging difficult. The correct default for a financial object is either None (explicit) or a well-known sentinel date.

Fix: Default to None and let callers provide the date explicitly: 

self.valdate = kwargs.get('valdate', dt.datetime.now())
self.valdate = kwargs.get('valdate', None)
