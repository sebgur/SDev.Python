#1 New variance swaps (start ≥ valdate) always crash — current_fixing never set High
sdevpy/montecarlo/payoffs/basic.py · lines 452, 496–501

Why it's bad: Variance.__init__ sets self.current_fixing = None. In set_valuation_date, when start ≥ valdate (a brand-new swap with no history), hist_fixing_dates is empty, hist_fixings returns None or [], and the condition len(hist_fixings) ≥ 1 is never satisfied — so current_fixing stays None. Then evaluate() raises on every pricing call: 

if self.current_fixing is None:
    raise ValueError(f"Fixing not set for variance on {self.name}")

The only test in test_mc.py uses start_date = 2025-11-15 and valdate = 2025-12-15, which always has historical fixings. The failing case (start = valdate, no history) is never covered.

Fix: When hist_fixing_dates is empty, seed current_fixing from the current spot, or expose a set_current_fixing(spot) method and document that it must be called before pricing a new contract: 

if hist_fixings is not None and len(hist_fixings) >= 1:
    self.current_fixing = hist_fixings[-1]
else:
    self.current_fixing = fxgs.get_spot(self.name)  # new contract: use today's spot
#2 Maximum and BarrierDown WIP stubs still exported in the public API Medium
sdevpy/montecarlo/payoffs/exotics.py · lines 71–96

Why it's bad: Both classes are tagged "Warning: work in progress" and raise NotImplementedError in evaluate(), yet they are importable from the public sdevpy.montecarlo.payoffs.exotics namespace. Maximum.__init__ sets self.eventdates at construction time (wrong lifecycle phase — this must be deferred to set_valuation_date). BarrierDown stores a raw integer index, bypassing the name-resolution system. #7 brownianbridge() outer factor loop is pure Python — easily vectorized Low
sdevpy/maths/rand/pathconstruction.py · lines 43–72

Why it's bad: The function iterates for d in range(n_factors) and runs the full midpoint-fill bridge construction for each factor column independently. For a 10-asset book this is 10 Python iterations of a nested while intervals loop. The terminal assignment and every bridge midpoint can be computed across all factors simultaneously by replacing the integer index d with a slice :: 

# Current — one factor column at a time:
for d in range(n_factors):
    w[:, -1, d] = np.sqrt(t) * z[:, 0, d]
    ...bridge fill for factor d...

# Vectorized — all factor columns at once:
w[:, -1, :] = np.sqrt(t) * z[:, 0, :]
...bridge fill across n_paths × n_factors simultaneously...

The while intervals loop itself is sequential (each midpoint depends on its neighbours), but its body can act on the full n_paths × n_factors slice once the outer factor loop is eliminated.

#8 check_global_params() not guarded for params=None — misleading crash Low
sdevpy/volatility/impliedvol/models/tssvi1.py · line 123  ·  tssvi2.py · line 117

Why it's bad: calculate() and dump_data() both raise a clear RuntimeError when self.params is None. check_params() calls check_global_params(), which immediately unpacks self.params without a guard. When params is None, the caller gets: 

TypeError: cannot unpack non-iterable NoneType object

In the normal optimiser flow update_params is always called first, so this is not a production risk. But the inconsistency is a trap for anyone who constructs a model and calls check_params() directly.

Fix: Add a one-line guard at the top of check_global_params() in each model: 

def check_global_params(self) -> tuple[bool, float]:
    if self.params is None:
        return False, constants.FLOAT_INFTY
    ...
#9 implied_vols() uses a serial Brent loop; the vectorized implied_vol_newton() is available but unused here Low
sdevpy/analytics/black.py · lines 51–60

Why it's bad: implied_vols() calls the scalar Brent solver once per (strike, price) pair. implied_vol_newton() (line 63) is a fully vectorized Newton–Raphson implementation that handles entire arrays in one call and converges faster. Every calibration pass through prepare_target_data calls implied_vols() for price-to-vol conversion, making this a serial per-option bottleneck. The Brent tolerance was improved from 1e-4 to 1e-6 but the structural inefficiency remains.

Fix: Replace the loop in implied_vols() with a single call to implied_vol_newton(). Retain the Brent path as a fallback for strikes where Newton fails to converge (very OTM, very short-dated), as the existing docstring already anticipates: 

ivs = [implied_vol(expiry, k, is_call, fwd, p) for k, p in zip(strike, fwd_price, strict=True)]
return np.asarray(ivs)
return implied_vol_newton(expiry, strike, is_call, fwd, fwd_price)
#10 Arithmetic payoff classes repeat identical lifecycle boilerplate across six implementations Low
sdevpy/montecarlo/payoffs/basic.py · lines 514–641 (Add, Sub, Mul, Div, Neg, Basket)

Why it's bad: Add, Sub, Mul, Div, Neg, and Basket each implement set_nameindexes, set_valuation_date, and set_eventindexes with nearly identical bodies: iterate over children, delegate the same method, then collect event dates via list_payoff_eventdates. The pattern repeats ~18 times across ~130 lines. Adding a new binary combinator (e.g. Pow) requires copying all four methods verbatim. Any future lifecycle change (a new hook) must be applied in every class.

Fix: Introduce a BinaryOp(Payoff) base class that holds self.left / self.right and implements the three shared lifecycle methods generically. Add, Sub, Mul, and Div then only need to override evaluate(). A parallel UnaryOp handles Neg and Abs. This collapses ~130 lines of boilerplate to ~30.
