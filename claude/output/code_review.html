#1 Copy-paste bug in biexp.py: left-branch guard checks taur == 0 instead of taul == 0 Critical
sdevpy/volatility/impliedvol/models/biexp.py · lines 91, 99

Why it's bad: The biexp() function has two zero-checks — one for the right branch (taur) and one for the left branch (taul). Both accidentally check the same variable: 

# Right case
if isequal(taur, 0.0):
    raise ValueError("taur=0 is not an acceptable parameter")
...
# Left case  --  BUG: checks taur again instead of taul
if isequal(taur, 0.0):
    raise ValueError("taul=0 is not an acceptable parameter")

When taul = 0 is passed in the left-branch guard is never triggered. The expression np.exp(xm_l / taul) on line 104 then produces inf or nan silently. The error message even says "taul=0" while testing taur, confirming a copy-paste error. It is not caught by biexp_check_params because that function only enforces taul > 0 as a soft constraint, not a hard zero-check.

Fix: Change line 99 from isequal(taur, 0.0) to isequal(taul, 0.0).

#2 LogMix.update_params() stores raw input — dump_data() crashes with AttributeError when called after initial_point() High
sdevpy/volatility/impliedvol/models/logmix.py · lines 256–259, 365–369

Why it's bad: LogMix.update_params() overrides the base class and stores self.params = x without coercing to ndarray: 

def update_params(self, x):
    self.params = x  # raw list stored; base class uses np.asarray(x, dtype=float)
    self.set_param_functions(self.params)

dump_data() then calls self.params.tolist(). When the caller passes a plain Python list — as returned by initial_point() on line 363 which does return init_params (a plain list) — this crashes with AttributeError: 'list' object has no attribute 'tolist'. This is the standard usage pattern in the codebase (call initial_point() then update_params() then dump()) and every other ParametricImpliedVol subclass inherits the coercing base implementation.

Fix: Either delete LogMix.update_params() entirely so the base implementation is inherited, or coerce explicitly: 

def update_params(self, x) -> None:
    self.params = np.asarray(x, dtype=float)
    self.set_param_functions(self.params)
#3 calib_lv_dupire leaves lv[0] and moneynesses[0] as None — crash inside MatrixLocalVol High
sdevpy/volatility/localvol/dupire_calib.py · lines 153–182

Why it's bad: The loop for i in range(0, n_times - 1) fills indices 0 through n_times − 2. The last slot is then patched to the penultimate: moneynesses[-1], lv[-1] = moneynesses[-2], lv[-2]. But the first slot was also supposed to be patched — the line # moneynesses[0], lv[0] = moneynesses[1], lv[1] is commented out. This leaves moneynesses[0] and lv[0] as None. These are passed directly to MatrixLocalVol(t_grid, np.log(moneynesses), lv), where np.log(None) raises a TypeError. The function is therefore broken for every standard time grid with two or more intervals.

Fix: Uncomment the first-slot patch (or simply handle it inside the loop when i == 0): 

# After the loop, patch both boundary slots:
moneynesses[0], lv[0] = moneynesses[1], lv[1]
moneynesses[-1], lv[-1] = moneynesses[-2], lv[-2]
#4 Variance payoff always raises at pricing when contract start ≥ valdate High
sdevpy/montecarlo/payoffs/basic.py · lines 452, 496–501, 463–464

Why it's bad: Variance.__init__ sets self.current_fixing = None. In set_valuation_date, if start ≥ valdate (a brand-new variance swap), all dates go into self.eventdates and hist_fixing_dates is empty. The guard on line 500 — if hist_fixings is not None and len(hist_fixings) ≥ 1 — is never satisfied, so current_fixing stays None. Then evaluate() unconditionally raises: 

if self.current_fixing is None:
    raise ValueError(f"Fixing not set for variance on {self.name}")

The only test in test_mc.py uses start_date = 2025-11-15 (one month before valdate = 2025-12-15), so historical fixings always exist and this failing case is never exercised.

Fix: When hist_fixing_dates is empty (new contract), initialise current_fixing from the current spot in market data, or expose a set_current_fixing(spot) method and document it as a pre-pricing requirement.

#5 Two live print() calls inside pytest test functions pollute CI output Medium
sdevpy/tests/test_impliedvol.py · line 190  ·  sdevpy/tests/test_localvol.py · line 144

Why it's bad: Both are inside actual def test_*() functions (not guarded by if __name__ == "__main__"), so they fire on every pytest run: 

# test_impliedvol.py:190  inside test_vsvi_formula()
print(test)

# test_localvol.py:144  inside test_lv_bysections_builder_calculate_vols_…()
print(vols)

In a CI pipeline the noise can obscure real failures. All other debug prints in the same files are correctly commented out — these two were missed.

Fix: Comment out both lines, matching the pattern already used in the same files.

#6 Hard-coded name_model_map in localvol_factory.py silently breaks for any real ticker Medium
sdevpy/volatility/localvol/localvol_factory.py · line 19

Why it's bad: The factory contains a module-level dict: 

name_model_map = {'ABC': 'BiExp', 'KLM': 'VSVI', 'XYZ': 'Matrix'}

Any call to load_param_lv(date, name) or get_local_vols(names, valdate) without an explicit model_name keyword falls back to this map. For any ticker outside these three test fixtures, model_name becomes None and the function raises ValueError: No model name specified for name: …. Since get_local_vols is called from price_book in the MC pricer, any production deployment pricing real instruments will fail immediately.

Fix: Remove the hard-coded map and require callers to always supply model_name explicitly, or drive the mapping from a runtime configuration file.

#7 Maximum and BarrierDown are WIP stubs exported publicly with no early failure in __init__ Medium
sdevpy/montecarlo/payoffs/exotics.py · lines 71–96

Why it's bad: Both classes are tagged "ToDo: [Warning] work in progress" and both raise NotImplementedError only in evaluate(). They are importable from the public sdevpy.montecarlo.payoffs.exotics namespace. Maximum.__init__ sets self.eventdates at construction time (wrong lifecycle — should be deferred to set_valuation_date). BarrierDown stores a raw integer index instead of a name, bypassing the name-resolution system entirely. A caller who accidentally instantiates either class gets a runtime error deep inside the pricing engine with no indication of why.

Fix: Until fully implemented, prefix with an underscore (_Maximum, _BarrierDown) to remove from the public API, or add raise NotImplementedError in __init__ itself for immediate and clearly labelled failure.

#8 black.implied_vol caps at xmax=2.0 — silently wrong for vols above 200% Medium
sdevpy/analytics/black.py · lines 35–48

Why it's bad: implied_vol minimizes squared error on the interval [1e-6, 2.0]. Any option whose true implied vol exceeds 200% — short-dated deep OTM options, high-vol names, or variance swap component inversions — returns the clipped value 2.0 without raising an error. This function is called from LvObjectiveBuilder.calculate_vols() and optionsurface.convert_option(), so calibration quality metrics can be silently wrong. A vectorized Newton-Raphson implementation (implied_vol_newton) already exists in the same file and handles this better. 

xmax = 2.0
xmax = 10.0  # Raise cap; or replace call with implied_vol_newton
#9 LvObjectiveBuilder.calculate_vols() compares an OptionType enum against raw integer literals Medium
sdevpy/volatility/localvol/lvsection_calib.py · lines 328–335

Why it's bad: self.option_type is set via string_to_optiontype(option_type) which returns an OptionType enum. The comparison then uses raw integer literals: 

if self.option_type == 0:   # fragile: relies on OptionType.CALL.value == 0
    pde_vols.append(black.implied_vol(expiry, k, True, self.fwd, p))
elif self.option_type == 1:  # fragile: relies on OptionType.PUT.value == 1
    pde_vols.append(black.implied_vol(expiry, k, False, self.fwd, p))

This works today because OptionType is an IntEnum. If it is ever changed to a plain Enum, both comparisons silently evaluate to False and every call and put is routed to the straddle branch, producing wrong implied vols with no error. The rest of the codebase (e.g. forwardpde.py) uses the correct case OptionType.CALL: match syntax.

Fix: Replace integer literals with enum members: 

from sdevpy.instruments.constants import OptionType
if self.option_type == OptionType.CALL:
    ...
elif self.option_type == OptionType.PUT:
    ...
#10 InterpolatedYieldCurve.set_data() uses the unsorted original dfs list in the zero-rate branch Low
sdevpy/market/yieldcurve.py · lines 69–79

Why it's bad: set_data() sorts pillars by date into self.dates and self.dfs. When building the zero-rate interpolation grid, the code re-uses the original unsorted parameter name dfs rather than the sorted self.dfs: 

if self.interp_var == YieldCurveVariable.ZERORATE:
    times = timegrids.model_time(self.valdate, self.dates)  # sorted ✓
    data_y = -np.log(dfs) / times  # original unsorted parameter ✗

If pillar dates are passed in non-chronological order, times is sorted but data_y retains the original order, producing a mismatched interpolation grid that returns wrong discount factors silently. The DISCOUNT and LOG_DISCOUNT branches correctly use ext_dfs (built from the sorted self.dfs); only the default ZERORATE branch is affected.

Fix: 

data_y = -np.log(dfs) / times
data_y = -np.log(self.dfs) / times
