Claude's Recommendations — sdevpy

Quantitative finance library  ·  Python  ·  2026-06-01  ·  Ranked by materiality

What this report covers

sdevpy is a capable personal quant library with clean mathematical foundations, solid PDE and MC engines, and a well-structured implied-vol model hierarchy. It is currently built and maintained as a personal research tool. Turning it into a marketable FinTech product requires changes at three levels: reliability and correctness (a buyer will not trust a library that crashes in edge cases), enterprise integration (market data, logging, configuration, and packaging must be production-grade), and commercial differentiation (documentation, benchmarks, API stability, and extensibility determine whether a FinTech firm chooses this over QuantLib, RiskEngine, or an in-house build).

The recommendations below are ranked by how much they move the needle toward commercial viability. Each entry describes the current state, the target state, the packages or patterns to adopt, and the realistic effort involved.

Tier 1 — Table Stakes for Commercial Sale
R1 Define a stable, versioned public API with semantic versioning Market-critical Effort: Medium

Current state: There is no __all__ in most modules, no documented stable surface, and no semver versioning policy. WIP classes (Maximum, BarrierDown) are importable alongside production classes with no distinction. The pyproject.toml version is not managed programmatically.

Why it matters: An enterprise buyer integrates your library into their pricing stack. Without a stable API contract and a deprecation policy, every upgrade is a risk. This is the single biggest trust signal in B2B FinTech software sales.

  • Add __all__ to every public module, explicitly listing what is supported.
  • Move WIP classes to sdevpy/_experimental/ or prefix them with _.
  • Adopt semantic versioning (1.0.0 → 1.1.0 → 2.0.0) and use bump2version or hatch-vcs to drive the version from git tags.
  • Write and publish a CHANGELOG.md with one entry per release.
  • Add a deprecation decorator so old APIs can be kept alive with a warning before removal.

R3 Abstract the market data layer behind a provider interface Market-critical Effort: High

Current state: Market data (spots, forward curves, vol surfaces, fixings) is loaded directly from CSV and JSON files in sdevpy/market/. Functions like get_spots, get_local_vols, and get_yieldcurve in mcpricer.py have file-path logic baked in. settings.py hard-codes WORKFOLDER = r"C:\temp\sdevpy".

Why it matters: No institutional client will pipe Bloomberg or Refinitiv data through CSV files. The market data layer is the primary integration point for enterprise sales. It must be an interface that clients implement against their own data infrastructure (databases, APIs, caches). 

# Target: abstract provider protocol
from typing import Protocol
from datetime import datetime

class MarketDataProvider(Protocol):
    def get_spot(self, name: str, valdate: datetime) -> float: ...
    def get_yield_curve(self, curve_id: str, valdate: datetime) -> YieldCurve: ...
    def get_vol_surface(self, name: str, valdate: datetime) -> ImpliedVol: ...
    def get_fixings(self, name: str, dates: list[datetime]) -> dict[datetime, float]: ...
  • Ship a FileSystemProvider (current behaviour) as the default implementation.
  • Provide an InMemoryProvider for testing that takes plain Python dicts.
  • Document the protocol so clients can implement a BloombergProvider or DatabaseProvider against their own stack.
  • Replace settings.WORKFOLDER with pydantic-settings reading from environment variables and a .env file.

R5 Enforce a uniform model interface contract via ABC or Protocol High Effort: Medium

Current state: FactorModel is an ABC but its contract is not enforced: MultiAssetHeston doesn't inherit from it, initial_state has different signatures across subclasses, and ParametricImpliedVol allows params = None until update_params() is called, causing opaque crashes. The payoff lifecycle (set_valuation_date → set_nameindexes → evaluate) is informal and not enforced.

Why it matters: A client extending your library to add custom models or payoffs must understand the exact contract. If the interface is implicit, they will make mistakes, blame the library, and not renew. Formal interfaces are the difference between a framework and a collection of scripts.

  • Define typing.Protocol classes for FactorModel, PayoffComponent, ImpliedVolModel, and LocalVolModel with exact signatures.
  • Add a _require_params() guard to ParametricImpliedVol and call it at the top of every calculate().
  • Formalize the payoff lifecycle as an ABC with abstract methods in the correct call order and @abstractmethod enforcement.
  • Write a model validation checklist (as a class method or mixin) that any new model author can call to verify compliance: MyModel.validate_interface().
Tier 2 — Production Robustness
R6 Replace scattered print() calls with structured logging throughout High Effort: Low

Current state: The library mixes bare print() calls in production functions (meanreversion.py:104, localvol_factory.py:171, tools.py:40, scalendar.py:208) with logging.getLogger(__name__) already correctly used in the majority of modules. Several production files also contain if __name__ == "__main__": print("Hello") stubs.

Why it matters: Library code that writes to stdout breaks any downstream application that captures or redirects stdout. Enterprise platforms (trading systems, risk engines) expect library log output to flow through the application's own logging infrastructure. This is a disqualifying issue for enterprise sales.

  • Replace all print() in production code with log.info(), log.warning(), or log.debug().
  • Consider adopting structlog for structured (JSON-serializable) log records — standard in cloud-native FinTech deployments.
  • Add a NullHandler at the library root (logging.getLogger('sdevpy').addHandler(logging.NullHandler())) so the library is silent by default until the application configures logging. This is the Python logging best practice for libraries.
  • Remove all if __name__ == "__main__": print("Hello") stubs; replace with meaningful examples or delete them.
structlog
R7 Adopt pydantic for data model validation across the library High Effort: Medium

Current state: Configuration objects use plain Python @dataclass (PdeConfig, McConfig) with no field validation. Market data is passed as plain dicts or keyword arguments. Input validity (e.g., vol > 0, expiry > valdate, strike > 0) is either absent or checked inconsistently deep inside computation.

Why it matters: Validation at the boundary catches configuration errors before they corrupt a book-level risk run. Pydantic models also auto-generate JSON schemas, making API contracts machine-readable — useful for integration testing and documentation. 

# Before (plain dataclass, no validation)
@dataclass
class PdeConfig:
    n_timesteps: int = 25
    n_meshes: int = 100

# After (pydantic, validated)
from pydantic import BaseModel, Field, field_validator

class PdeConfig(BaseModel):
    n_timesteps: int = Field(25, gt=0, description="Number of PDE time steps")
    n_meshes:    int = Field(100, gt=0, description="Number of spatial mesh points")

    @field_validator('scheme')
    def validate_scheme(cls, v):
        allowed = {'Implicit', 'CN', 'Theta', 'Explicit', 'Rannacher'}
        if v not in allowed:
            raise ValueError(f"scheme must be one of {allowed}")
        return v

R8 Build a library-specific exception hierarchy High Effort: Low

Current state: Exceptions are a mix of ValueError, RuntimeError, TypeError, AttributeError, and NotImplementedError with no consistent hierarchy. A caller has no way to distinguish a calibration failure from a model configuration error from a market data problem.

Why it matters: In production systems, different exception types trigger different recovery actions: a calibration failure might fall back to the previous day's parameters; a market data error might trigger an alert; a pricing error might suppress the trade. Without a typed exception hierarchy, all of these look identical to the caller. 

# sdevpy/exceptions.py
class SdevpyError(Exception): """Base for all sdevpy errors"""

class MarketDataError(SdevpyError): """Market data missing or invalid"""
class CalibrationError(SdevpyError): """Calibration failed to converge"""
class ModelError(SdevpyError):       """Model configuration or state error"""
class PricingError(SdevpyError):     """Error during payoff evaluation"""
class ArbitrageError(ModelError):    """Model violates no-arbitrage conditions"""

Replace all generic exceptions that cross module or public-API boundaries with the appropriate subclass. Keep built-ins for internal assertions and programming errors.

R9 Add a model reproducibility and audit trail framework Market-critical Effort: Medium

Current state: There is no formal mechanism for capturing the exact inputs, random seed, model parameters, and library version used for a given pricing run. The MC pricer does not log or persist what was computed or with what configuration.

Why it matters: Regulatory requirements (FRTB, Dodd-Frank, MiFID II model risk governance) require institutions to demonstrate that a model can reproduce a given result. This is a hard requirement for any risk or pricing model used in a regulated entity. Without it, you cannot be used in a bank.

  • Add a PricingContext object that records: library version, git commit hash, model names and serialized parameters, random seed, valuation date, number of paths, and timings. Emit it as a structured log record at the end of each price_book call.
  • Use hashlib to fingerprint input market data and model parameters, so results can be reproduced from the fingerprint alone.
  • Expose a price_book(..., seed=42) parameter that seeds all RNGs deterministically and documents this in the API.
  • Consider integrating mlflow or a lightweight equivalent for parameter tracking across calibration runs.

Tier 3 — Architecture and Performance
R11 Parallelize Monte Carlo path generation with joblib or Ray Performance Effort: Medium

Current state: PathGenerator.generate_paths and the MC pricer are single-threaded. For a book of 100 trades with 100,000 paths each, the runtime is proportional to n_trades × n_paths × n_timesteps in a single thread.

Why it matters: Production risk systems must price books in seconds, not minutes. Parallelization is the fastest way to cut wall-clock time. Buyers evaluate performance benchmarks before committing to a library.

  • Split the path batch into n_workers sub-batches and aggregate: joblib.Parallel(n_jobs=-1)(generate_batch(seed+i) for i in range(n_workers)).
  • For very large books, provide a Ray or Dask backend where each trade is priced as a remote task.
  • Alternatively, use numpy's numba JIT on the innermost path-stepping loops for single-threaded speedup without architecture changes.
  • Expose a simple n_workers parameter on McConfig with a default of 1 for backwards compatibility.
joblibraynumbadask
R12 Accelerate tree roll-back with vectorized NumPy slices Performance Effort: Low

Current state: Both BinomialTree.roll_back and TrinomialTree.roll_back in trees.py build the continuation value vector via Python list comprehensions — the same O(n2) anti-pattern that was correctly fixed in the PDE schemes in the last review cycle. 

# Current — Python list comprehension, O(n²) total
cont_v = np.asarray([df * (p * v[i + 1] + (1.0 - p) * v[i]) for i in range(size)])
# Vectorized — single C-speed operation
cont_v = df * (p * v[1:size+1] + (1.0 - p) * v[:size])

The trinomial version is analogously replaceable. For a 200-step tree, this is a ~100× speedup on the inner loop with zero architectural change.

R14 Split the library into focused, independently installable packages Medium Effort: High

Current state: sdevpy is a single monolithic package mixing quantitative analytics, Monte Carlo, PDE, implied-vol calibration, local-vol calibration, tree pricing, ML/LLM tooling, and time-series analysis. All dependencies (TensorFlow, Keras, transformers) must be installed even if a client only needs Black-Scholes pricing.

Why it matters: Enterprise buyers want minimal footprint. A derivatives desk does not want to install TensorFlow as a transitive dependency of a pricing library. Modular packaging also allows independent versioning and testing.

  • sdevpy-core: analytics (Black, Bachelier), market data interfaces, utilities. Depends only on numpy/scipy.
  • sdevpy-derivatives: MC engine, PDE, tree, payoffs, vol models. Depends on sdevpy-core.
  • sdevpy-calibration: implied-vol and local-vol calibration. Depends on sdevpy-derivatives.
  • sdevpy-ml: ML surface generation, LLM tooling. Depends on sdevpy-core + optional ML deps.
  • Use a monorepo layout (one git repo, multiple packages under packages/) managed with hatch or uv workspaces.

R15 Formalize calibration as a pipeline with observable state Medium Effort: Medium

Current state: TsIvCalibrator.calibrate is a monolithic function. The calibration loop is not interruptible, does not emit progress events, and convergence diagnostics are only available by reading optimizer result objects after the fact.

Why it matters: Production calibration systems need to: report progress to a dashboard, apply time limits per expiry, fall back to a prior-day surface on failure, and expose the RMSE curve for model-risk governance. None of this is possible with the current opaque loop.

  • Introduce a CalibrationCallback interface with hooks: on_expiry_start, on_expiry_end(rmse, params), on_failure(expiry, error).
  • Add a CalibrationResult dataclass that captures per-expiry RMSE, iteration count, wall-clock time, and whether the solver converged.
  • Add a time limit per expiry via maxiter and a fallback strategy parameter (keep previous params vs. raise).
  • Expose a warmstart option that initializes each expiry from the previous day's calibrated parameters.
Tier 4 — Incremental Quality Improvements
R16 Replace minimize_scalar(method='brent') with a bounded solver across analytics Correctness Effort: Low

Current state: black.implied_vol and bachelier.implied_vol_solve use minimize_scalar(method='brent', bracket=(xmin, xmax)). Brent's method with bracket= does not constrain the search to [xmin, xmax] and can return negative volatility for out-of-model prices.

Fix: Switch to method='bounded' with bounds=(xmin, xmax), or replace the scalar Brent solver with the already-present vectorized Newton-Raphson (implied_vol_newton) which is both faster and more accurate. Add a post-solve sanity check that raises CalibrationError if the returned vol is outside a reasonable range.

R17 Add property-based tests for mathematical identities using Hypothesis Testing Effort: Low

The current test suite uses fixed regression values. Property-based testing with Hypothesis generates random inputs and verifies structural properties, finding edge cases that regression tests miss. 

from hypothesis import given, settings
from hypothesis.strategies import floats

@given(
    fwd=floats(min_value=50, max_value=200),
    strike=floats(min_value=10, max_value=500),
    vol=floats(min_value=0.01, max_value=2.0),
    expiry=floats(min_value=0.01, max_value=5.0)
)
def test_put_call_parity(fwd, strike, vol, expiry):
    call = black.price(expiry, strike, True,  fwd, vol)
    put  = black.price(expiry, strike, False, fwd, vol)
    assert np.isclose(call - put, fwd - strike, rtol=1e-10)

R18 Add correlation and vol surface input validation at market data boundaries Correctness Effort: Low

Current state: Correlations loaded from CSV are not validated to lie in [-1, 1], and the matrix is not checked for positive semi-definiteness before the Cholesky decomposition. A typo (1.5 instead of 0.5) propagates silently until simulation time. Similarly, vol surface sections are not checked for per-expiry strike/vol count consistency.

Add validation in correlations.py: check all values in [-1, 1+ε], then attempt Cholesky and wrap the LinAlgError in a descriptive MarketDataError with the offending matrix in the message.

R19 Standardize the test tolerance convention and tighten loose assertions Testing Effort: Low

Current state: ~30 test call sites use np.allclose(a, b, 1e-10), which sets rtol=1e-10 not atol. One call-put parity test uses atol=0.2, giving essentially no quantitative guarantee. Tolerance semantics are inconsistent across the test suite.

Define a project-wide assert_close(a, b, atol, msg="") helper in a tests/utils.py module and replace all np.allclose calls with it. This makes intent explicit, allows consistent error messages, and is a one-line change per test site.
