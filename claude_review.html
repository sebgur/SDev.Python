#1 MultiOptimizer.minimize() returns (result, nfev) — breaks the Optimizer interface High
sdevpy/maths/optimization.py · lines 99–114

Why it's bad: SciPyOptimizer.minimize() returns a single result object. MultiOptimizer.minimize() returns (result, nfev) — a tuple. Any caller that treats the two implementations interchangeably (which is the whole point of the Optimizer base class) will receive a tuple instead of a result and immediately fail when accessing result.x or result.fun. The abstract base method signature on line 38 also declares args as a required positional parameter, while both concrete implementations make it optional — a further contract violation.

Fix: Make MultiOptimizer.minimize() return just result, store nfev as an instance attribute if needed:

def minimize(self, f, x0=None, args=(), bounds=None):
    result = None
    self.total_nfev = 0
    for i, optimizer in enumerate(self.optimizers_):
        result = optimizer.minimize(f, x0, args, bounds)
        self.total_nfev += result.nfev
        if result.fun < self.mtol_:
            break
    return result  # not (result, nfev)

#3 Optimizer base class does not inherit ABC — @abstractmethod has no effect Medium
sdevpy/maths/optimization.py · line 36

Why it's bad: class Optimizer: declares minimize() as @abstractmethod but does not inherit from ABC. Python only enforces abstract method contracts when the class (or a parent) is an instance of ABCMeta. As written, you can instantiate Optimizer() directly and call minimize() — it would return None silently. New subclasses that forget to implement minimize() will also instantiate without error.

Fix:

from abc import ABC, abstractmethod

class Optimizer:
class Optimizer(ABC):
    @abstractmethod
    def minimize(self, f, x0=None, args=(), bounds=None):
        """ Minimization """
#4 RuntimeError used for wrong-count and invalid-parameter checks in svivol.py and biexp.py Medium
sdevpy/models/svivol.py · lines 55, 67  ·  sdevpy/models/biexp.py · lines 70, 83

Why it's bad: The identical issue was fixed in tssvi1.py in the previous cycle, but the same pattern persists in svivol.py and biexp.py. Python convention (PEP 3151) reserves RuntimeError for errors that don't fit other categories. Wrong parameter count and invalid parameter values are structural input problems — both should raise ValueError. Callers catching ValueError for bad-input recovery will silently miss these failures.

Fix (same pattern as tssvi1.py):

# svivol.py line 55:
raise RuntimeError(f"Incorrect parameter size in SviVol: {len(params)}")
raise ValueError(f"Expected 5 parameters for SviVol, got {len(params)}")

# svivol.py line 67 and biexp.py line 83:
raise RuntimeError("Invalid SviVol parameters")
raise ValueError("Invalid SviVol parameters")
#5 TermStructureParametricZeroSurface.calibrate_modelled_type() always raises — standard calibration interface unusable Medium
sdevpy/models/surfaces/zerosurface.py · line 282

Why it's bad: The method unconditionally raises NotImplementedError("Not implemented yet: calibrate_modelled_type"). This means calling surface.calibrate() on any TsSvi1 object will always fail at the last step, after all the consistency checks and data preparation have already run. The ZeroSurface.calibrate() → calibrate_modelled_type() contract is completely broken for the entire TermStructure* family. Calibration currently only works via the separate TsSvi1ObjectiveBuilder path, which bypasses the standard interface entirely. This is a hidden architectural gap that will bite any future developer who follows the documented interface.

Fix: Either override calibrate_modelled_type() in TsSvi1 to run the optimizer, or make TermStructureParametricZeroSurface.calibrate_modelled_type() call calibrate_tssvi1(). At minimum, document the design decision so the split is intentional and visible.

#9 TsSvi1.__main__ reconstructs model_vols using strikes from the last loop iteration Medium
sdevpy/models/surfaces/tssvi1.py · lines 194–199

Why it's bad: Lines 148–158 build flat arrays by iterating over expiries, assigning strikes = strike_surface[i] each iteration. After the loop, strikes refers to the last expiry's strike list. The reconstruction loop at lines 194–199 then uses this stale reference: 

for _ in expiries:
    vols = []
    for _ in strikes:  # always uses strike_surface[-1]
        vols.append(z[counter])
        counter += 1
    model_vols.append(vols)

If expiries have different numbers of strikes, the counter will be wrong for all but the last expiry — producing incorrect or misaligned model_vols and ultimately a wrong RMSE chart. The plotting loop at line 216 then computes rmse(vol_surface[exp_idx], model_vols[exp_idx]) on mis-sized arrays.

Fix: Track the strike count per expiry explicitly:

counter = 0
for exp_idx, _ in enumerate(expiries):
    n = len(strike_surface[exp_idx])
    model_vols.append(z[counter:counter + n].tolist())
    counter += n
#10 No calibration smoke test — attribute/method naming bugs will keep reaching production undetected Medium
sdevpy/tests/ · codebase-wide

Why it's bad: The check_degrees_of_freedom shadow (Critical, previous review) would have been caught in milliseconds by a single test that calls ZeroSurface.check_consistency() on any concrete subclass. The MultiOptimizer interface break (issue #1 above) is only detectable by a test that switches between optimizer types. Without a calibration smoke test, the same class of bugs will recur on every change to the calibration path.

Fix: Add at minimum: (1) a test_optionsurface.py round-trip conversion test verifying that convert_option() is the inverse of itself; (2) a smoke test that constructs a TsSvi1 and calls check_consistency() on a minimal option grid to verify the lifecycle; (3) a MultiOptimizer test that checks result.x is accessible on the returned value.
