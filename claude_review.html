#1 check_degrees_of_freedom boolean attribute shadows the method of the same name — TypeError on every calibration call Critical
sdevpy/models/surfaces/zerosurface.py · lines 24, 218–219, 224

Why it's bad: __init__ sets self.check_degrees_of_freedom = True (line 24), an instance attribute that shadows the method def check_degrees_of_freedom(self, options) defined at line 224. In check_consistency() lines 218–219: 

if self.check_degrees_of_freedom:        # evaluates the boolean True → truthy
    self.check_degrees_of_freedom(c_options) # calls True(...) → TypeError!

Python raises TypeError: 'bool' object is not callable on every calibration call where check_degrees_of_freedom is True. This is the same attribute/method shadowing pattern that previously affected expiry_times.

Fix: Use a distinct name for the flag and the method:

# In __init__:
self.check_degrees_of_freedom = True
self.do_check_degrees_of_freedom = True

# In check_consistency():
if self.check_degrees_of_freedom:
    self.check_degrees_of_freedom(c_options)
if self.do_check_degrees_of_freedom:
    self.check_degrees_of_freedom(c_options)
#2 Debug print() statements left in TsSvi1.formula() and formula_parameters() High
sdevpy/models/surfaces/tssvi1.py · lines 23–25, 67

Why it's bad: Four print() calls fire on every option evaluation. For a vol surface queried across a grid of strikes and expiries — or repeatedly during calibration — this floods stdout with thousands of lines, degrades performance, and breaks any pipeline that parses or captures output. Line 67 also prints the internal svi_params array that is immediately discarded (the function returns the list on line 68, making the array redundant).

Fix: Remove all four lines. If diagnostic output is needed during development, replace with logging.debug():

print(f"time: {t}")
print(f"x: {log_m}")
print(f"params: {params}")
# and
svi_params = np.asarray([a, b, r, m, s])
print(f"shape inside: {svi_params.shape}")
#3 WorstOfBarrier missing set_valuation_date and set_eventindexes High
sdevpy/montecarlo/payoffs/exotics.py · lines 39–51

Why it's bad: Both lifecycle methods are commented out with # ToDo: implement?. The base-class no-ops fire instead, so self.eventdates is never populated and evaluate() operates on whatever path layout happens to be present. The expiry is hardcoded to [:, -1, :] (last event index), which is wrong whenever the book contains trades with later maturities. This produces silently wrong prices with no error.

Fix: Implement both methods following the Terminal pattern: accept an expiry date, register it in set_valuation_date, resolve the grid index in set_eventindexes, and use self.expiry_idx in evaluate().

#4 WorstOf.evaluate() hardcodes the last event index — wrong for non-terminal expiries High
sdevpy/montecarlo/payoffs/basic.py · line 411

Why it's bad: spot_all[:, -1, :] always picks the last position in the event grid, which only coincides with the trade's expiry when it is the longest-dated trade in the book. In a multi-trade portfolio any shorter-dated WorstOf is evaluated at the wrong date, silently producing wrong PVs. WorstOf also has no set_valuation_date or set_eventindexes, so there is no mechanism to correct this.

Fix: Accept an expiry, resolve the index via set_eventindexes, and use it:

spot_all_at_exp = spot_all[:, -1self.expiry_idx, :]

#7 RuntimeError used for wrong parameter count — should be ValueError Medium
sdevpy/models/surfaces/tssvi1.py · line 73

Why it's bad: Python convention is that ValueError signals a value (including container size) that is structurally valid but wrong in content. RuntimeError is a last resort for errors that don't fit elsewhere. Using the wrong exception type means callers that catch ValueError for bad-input recovery will silently miss this failure. The codebase already uses ValueError correctly for size checks elsewhere (e.g. svi.py).

Fix:

raise RuntimeError("The number of parameters should be 11 for TsSvi1")
raise ValueError(f"Expected 11 parameters for TsSvi1, got {len(x)}")

#9 Maximum and BarrierDown are incomplete WIP classes exposed in the public API Medium

Why it's bad: Both classes are marked "Warning: work in progress", implement only evaluate(), and skip all lifecycle hooks (set_nameindexes, set_valuation_date, set_eventindexes).
