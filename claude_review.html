#1 get_strikes() silently returns None for any unrecognised type High
sdevpy/market/eqvolsurface.py · lines 46–51

Why it's bad: The method has two branches ('absolute' and 'relative') but no else clause. A typo such as get_strikes('Absolute') (note: the .lower() normalises case, so this particular typo is handled) or a genuinely unknown type like get_strikes('moneyness') falls off the end and returns None silently. The caller receives None where a list of arrays is expected, and the failure manifests several frames later as an unhelpful AttributeError: 'NoneType' object has no attribute '__len__' with no indication that a bad type string was passed. 

def get_strikes(self, type: str='absolute'):
    req_type = type.lower()
    if req_type == 'absolute':
        return self.abs_strikes
    elif req_type == 'relative':
        return self.abs_strikes / self.forwards.reshape(-1, 1)
    # ← falls off the end: returns None silently
    else:
        raise ValueError(f"Unknown strike type: {type!r}. Expected 'absolute' or 'relative'")
#2 dump_data() embeds np.float64 directly in a JSON dict — crashes on NumPy 2.x High
sdevpy/market/eqvolsurface.py · line 62

Why it's bad: self.forwards[i] is np.float64. In NumPy 1.x, np.float64 subclassed Python float and was transparently JSON-serialisable. In NumPy 2.x (confirmed: 2.3.5 in this project), that inheritance was removed. Calling json.dump(data, f) now raises TypeError: Object of type float64 is not JSON serializable on every call to dump(), making the surface persistence feature completely broken at runtime. 

# Current — crashes on NumPy 2.x:
section = {'expiry': expiry_str, 'forward': self.forwards[i], ...}

# Fix — explicit cast to native float:
section = {'expiry': expiry_str, 'forward': float(self.forwards[i]), ...}

Note: self.input_strikes[i].tolist() and self.vols[i].tolist() are already correct — ndarray.tolist() converts to native Python types. Only the scalar forward needs the explicit float() cast.

#4 TsIvCalibrator.check_consistency() is implemented but never called — pre-calibration validation is silently skipped Medium
sdevpy/volatility/impliedvol/impliedvol_calib.py · lines 51–76, 107–125

Why it's bad: check_consistency() (lines 107–125) filters negative-rate options, validates expiry/forward consistency, converts to the model's target quote type, and checks degrees of freedom. The docstring explicitly states it "is not used yet". The calibrate() method calls prepare_target_data() directly and goes straight to the optimizer, bypassing all of these guards. This means: (a) options with non-positive forwards or strikes are passed silently to the model; (b) the DOF check (self.model.check_dof) never fires; (c) the quote-type conversion logic in convert_to_target_values() is dead code from the calibrator's perspective.

Fix: Either wire check_consistency() into calibrate() for the OptionTarget-based path (when quote-type conversion is needed), or add the relevant guards (positive-forward filter, DOF check) inline in calibrate() and remove the unused method.

#6 Hardcoded infeasibility penalty 100.0 * market_vols.sum() ignores the model's own penalty value Medium
sdevpy/volatility/impliedvol/impliedvol_calib.py · line 40

Why it's bad: When check_params() returns is_ok=False, the objective returns a magic number (100.0 * self.market_vols.sum()) that is unrelated to the model's own penalty (second element of the tuple, currently constants.FLOAT_INFTY in TsSvi1). If the sum of market vols is small — for example, near-zero vols in a normalised dataset — the infeasibility penalty could be smaller than the objective at any valid solution, causing the optimizer to converge into an infeasible region with no warning.

#10 WorstOfBarrier and WorstOf payoffs remain incomplete — silently produce wrong prices Medium
sdevpy/montecarlo/payoffs/exotics.py · lines 9–51  ·  sdevpy/montecarlo/payoffs/basic.py · line 403

Why it's bad: WorstOfBarrier has no set_valuation_date() or set_eventindexes() — the base-class no-ops fire and self.eventdates is never populated. WorstOf.evaluate() still hardcodes spot_all[:, -1, :] (the last path step), which only coincides with the correct expiry when the trade is the longest-dated in the portfolio. In a multi-trade book, any shorter-dated WorstOf is evaluated at the wrong date, silently producing incorrect PVs. This has been noted in prior reviews and remains unresolved.

Fix: Implement set_valuation_date() and set_eventindexes() in both payoffs following the Terminal pattern, store the resolved index, and replace -1 with it in evaluate(). Alternatively, guard with raise NotImplementedError to make failures explicit.
