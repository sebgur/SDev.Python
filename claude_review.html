#1 Calibration infeasibility penalty 100.0 * target_values.sum() is scale-dependent Medium
sdevpy/volatility/impliedvol/impliedvol_calib.py · line 51

Why it's bad: The penalty scales with the number of options and the magnitude of target values. For a dense surface with large prices the penalty can exceed any feasible RMSE by orders of magnitude, creating a near-vertical gradient at the boundary that prevents the optimizer recovering into the feasible region. The inline comment acknowledges this uncertainty but leaves it open across multiple reviews.

Fix: Use the penalty already returned by check_params():

return 100.0 * self.target_values.sum()
return penalty  # constants.FLOAT_INFTY from model.check_params()
#2 npt.ArrayLike used as return type — it is an input-only protocol Medium
sdevpy/analytics/black.py · lines 11, 37, 49  ·  sdevpy/analytics/bachelier.py · lines 9, 19

Why it's bad: npt.ArrayLike is defined as a union of things that can be converted to an ndarray — lists, tuples, scalars, arrays. Using it as a return type tells mypy and IDEs the function might return a list or a tuple, which is wrong. Every function here returns a concrete ndarray. Callers that do .shape or .dtype on the return value will see no static error even if the value is a list.

Fix:

def price(...) -> npt.ArrayLike:
def price(...) -> npt.NDArray[np.float64]:

Applies to black.price(), black.implied_vols(), black.implied_vol_newton(), bachelier.price(), bachelier.price_straddles().

#3 implied_vol_newton() returns a scalar or an array depending on input size Medium
sdevpy/analytics/black.py · line 73

Why it's bad: return (vol.item() if vol.ndim == 0 or vol.size == 1 else vol) means the return type changes shape depending on what was passed in. A 1-element array input returns a Python float; a 2-element input returns an ndarray. Callers cannot write generic code that works uniformly — any code that calls .shape on the result will crash for the scalar case, and code expecting a scalar will get an array for the 2-element case. The annotation -> npt.ArrayLike does not document this split.

Fix: Always return npt.NDArray[np.float64]. Drop the special-case entirely. Callers that need a Python scalar can call .item() themselves. 

return (vol.item() if vol.ndim == 0 or vol.size == 1 else vol)
return vol
#4 bachelier.price_straddles() has no test despite complex 2-D indexing Medium
sdevpy/analytics/bachelier.py · lines 18–34  ·  sdevpy/tests/test_analytics.py

Why it's bad: price_straddles() has a double loop, 2-D vol[i, j] indexing, a reshape(-1, 1), and relies on call_price[0] index access — more moving parts than any other function in the analytics module. Every other Bachelier function has a round-trip test; this one has none. A silent regression (e.g. wrong axis, off-by-one index) would go undetected.

Suggested test:

def test_bachelier_price_straddles():
    fwd = 0.04
    expiries = np.array([0.5, 1.0])
    strikes  = [np.array([0.03, 0.04, 0.05]),
                np.array([0.03, 0.04, 0.05])]
    vols = np.full((2, 3), 0.05)
    result = bachelier.price_straddles(expiries, strikes, fwd, vols)
    # straddle == call + put; verify against price() directly
    call = bachelier.price(0.5, 0.03, True,  fwd, 0.05)
    put  = bachelier.price(0.5, 0.03, False, fwd, 0.05)
    assert np.isclose(result[0, 0], call + put)
#5 price_straddles() crashes on malformed input with no helpful error message Medium
sdevpy/analytics/bachelier.py · lines 27–28

Why it's bad: strike[i] assumes strike is a jagged list of arrays (one per expiry), and vol[i, j] assumes a 2-D numpy array. Passing a flat 1-D array for either produces an IndexError or TypeError with a traceback pointing into the loop — not at the caller's input. The npt.ArrayLike annotation gives no indication of these constraints.

Fix: Add an early guard and document the expected shapes in the docstring:

vol = np.atleast_2d(np.asarray(vol, dtype=float))
n_exp = len(expiries_)

    raise ValueError(f"vol must have {n_exp} rows, got {vol.shape[0]}")

#10 No test for LogMix.pdf() or LogMix.cdf() Low
sdevpy/volatility/impliedvol/models/logmix.py · lines 201–238  ·  sdevpy/tests/test_impliedvol.py

Why it's bad: pdf() and cdf() implement non-trivial weighted sums of log-normal densities with shift and scale corrections. They are untested. A basic sanity check — that the PDF integrates to approximately 1 and the CDF is monotone — would catch any regression in the weight/normalization logic.

Suggested tests:

def test_logmix_pdf_integrates_to_one():
    model = LogMix(2)
    model.update_params(model.initial_point())
    t, fwd = 1.0, 100.0
    strikes = np.linspace(50, 200, 500)
    pdf_vals = np.array([model.pdf(t, k, fwd) for k in strikes])
    integral = np.trapz(pdf_vals, strikes)
    assert np.isclose(integral, 1.0, atol=0.01)

def test_logmix_cdf_monotone():
    model = LogMix(2)
    model.update_params(model.initial_point())
    t, fwd = 1.0, 100.0
    strikes = np.linspace(60, 160, 50)
    cdf_vals = np.array([model.cdf(t, k, fwd) for k in strikes])
    assert np.all(np.diff(cdf_vals) >= 0)
