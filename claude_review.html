#1 TermStructureParametricZeroSurface.calibrate_modelled_type() always raises — TsSvi1.calibrate() is permanently broken Critical
sdevpy/volatility/impliedvol/zerosurface.py · line 280

Why it's bad: The abstract contract requires subclasses to implement calibrate_modelled_type(). TermStructureParametricZeroSurface provides a default that always raises: 

def calibrate_modelled_type(self, date, options):
    raise NotImplementedError("Not implemented yet: calibrate_modelled_type")

TsSvi1 inherits this class and does not override calibrate_modelled_type(). Calling TsSvi1().calibrate() always raises at runtime despite the class having all other calibration pieces (formula_parameters(), bounds(), initial_point(), check_params()) fully implemented. There is no static or documented signal that the standard .calibrate() entry point is unavailable.

Fix: Either implement calibrate_modelled_type() in TsSvi1 using the existing TsSvi1ObjectiveBuilder pattern, or mark the method as intentionally unsupported with a docstring that directs users to the builder pattern. At minimum, make the base class method abstract so Python's ABC machinery raises at class definition time, not at runtime: 

def calibrate_modelled_type(self, date, options):
    raise NotImplementedError("Not implemented yet: calibrate_modelled_type")

@abstractmethod
def calibrate_modelled_type(self, date, options):
    """ Implement in subclass. See TsSvi1ObjectiveBuilder for the recommended pattern. """
    pass
#2 svi_check_params() and gsvi_check_params() annotated -> None but return (bool, float) tuples High
sdevpy/volatility/impliedvol/models/svi.py · line 53  ·  sdevpy/volatility/impliedvol/models/gsvi.py · line 35

#3 upper_bound() passes atol as rtol to np.isclose — wrong tolerance axis High
sdevpy/tools/algos.py · line 30

Why it's bad: np.isclose(a, b, rtol, atol) — the third positional argument is rtol (relative tolerance), not atol. The function signature is upper_bound(arr_sorted, value, atol=1e-11), so the intended absolute tolerance is silently used as the relative tolerance, while np.isclose's default atol=1e-8 applies unchanged: 

# Actual comparison for value ≈ 1.0:
if np.isclose(arr_sorted[idx - 1], value, atol):  # atol=1e-11 used as rtol!
# Effective test: |a - b| <= 1e-11 * |b| + 1e-8  ≈  1e-8  (not 1e-11)

if np.isclose(arr_sorted[idx - 1], value, rtol=0.0, atol=atol):
# Effective test: |a - b| <= 0.0 * |b| + 1e-11  =  1e-11  (correct)

The effective tolerance (~1e-8) is ~1000× more permissive than intended for model times around 1.0. This function is called in localvol.py (vol section lookup) and timegrids.py (date-to-index resolution). For typical time grids the minimum spacing is ~1/365 ≈ 2.7e-3, so the bug does not currently cause wrong section selection in practice — but it means the atol parameter has no effect and any future usage with fine grids could produce silent off-by-one errors.

Fix:

if np.isclose(arr_sorted[idx - 1], value, atol):
if np.isclose(arr_sorted[idx - 1], value, rtol=0.0, atol=atol):

#5 Production print() statements in SabrGenerator.generate_samples() Medium
sdevpy/volsurfacegen/sabrgenerator.py · lines 26–29, 33, 51

Why it's bad: Six print() calls are in generate_samples(), a core production method. Line 51 fires once per surface inside the inner loop: 

for j in range(num_surfaces):
    print(f"Surface generation number {j+1:,}/{num_surfaces:,}")  # once per surface

For a 50k-sample dataset with a 10×15 surface grid (150 points per surface), num_surfaces ≈ 333 — 333 print calls flood stdout, break any pipeline capturing output, and degrade performance. The header prints at lines 26–29 also expose internal configuration on every call.

#6 ParametricZeroSurface.formula() abstract method annotated -> float but all implementations return npt.ArrayLike Medium
sdevpy/volatility/impliedvol/zerosurface.py · line 261

Why it's bad: The abstract method declares a scalar return type: 

@abstractmethod
def formula(self, t: float, k: float, is_call: bool, f: float,
           params: list[float]) -> float:

Every concrete implementation (TsSvi1, SviSection, BiExp, CubicVol, etc.) accepts k: npt.ArrayLike and returns a NumPy array when k is an array. Any code path that type-checks the result as float — including downstream calculations that use scalar operations on the return value — is silently incorrect. The correct return hint is npt.ArrayLike or float | npt.NDArray.

Fix:

def formula(self, t: float, k: float, is_call: bool, f: float,
            params: list[float]) -> float:
def formula(self, t: float, k: npt.ArrayLike, is_call: bool, f: npt.ArrayLike,
            params: list[float]) -> npt.ArrayLike:

#8 localvol_factory.py raises TypeError for wrong input values — should be ValueError Medium
sdevpy/volatility/localvol/localvol_factory.py · lines 34, 44

Why it's bad: Python convention: TypeError means an argument has the wrong type; ValueError means it has the wrong value. Both raises in this file guard against missing or unknown string values — wrong values, not wrong types: 

# Line 34 — missing required keys (wrong value, not wrong type):
raise TypeError("Invalid section input in local vol file")
raise ValueError("Invalid section config: 'time' and 'model' are required")

# Line 44 — unknown model name string (wrong value, not wrong type):
raise TypeError(f"Unknown section type: {model}")
raise ValueError(f"Unknown local vol model: '{model}'. Expected one of: biexp, cubicvol, vsvi")

Using the wrong exception type means callers that catch ValueError for bad-input recovery will silently miss these failures.

#9 training.py uses torch.no_grad() and torch.nn.functional via implicit submodule import Low
sdevpy/machinelearning/llms/training.py · lines 1, 55, 67, 98

Why it's bad: The file only imports torch.nn (line 1), but uses bare torch directly on three lines: 

import torch.nn  # only torch.nn is imported explicitly

with torch.no_grad(): # line 55 — relies on implicit torch namespace
with torch.no_grad(): # line 67
loss = torch.nn.functional.cross_entropy(...)  # line 98

In CPython, import torch.nn does make torch available as a side effect of package import — so the code works today. But this is implicit, non-obvious behavior. Any refactoring that reorganises the import block could break all three call sites without a clear error message.

Fix:

import torch.nn
import torch
import torch.nn
