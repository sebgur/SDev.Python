#### Claude's recommendations on Architecture ####
The architecture is penalized mainly in four areas.

1. Separate calibration from model definition (biggest gain)

ZeroSurface is simultaneously a pricing model, a vol-conversion engine, and a calibrator.
The calibration concern — optimizer, stopping criteria, degrees-of-freedom check — belongs in a
dedicated Calibrator class that wraps a surface. The surface should just define calculate().
This pattern is common in quant libraries and makes it possible to swap optimizers without
touching the model.

2. Use the existing MC framework consistently

fbsabr.py, mcheston.py, and mczabr.py each hand-roll their own Monte Carlo loop
(draw gaussians → evolve → collect payoff), duplicating the logic that already lives in
PathGenerator. The architecture has the right abstractions — Payoff, PathGenerator, MarketState —
but the analytics modules bypass them entirely. Unifying them would eliminate the duplication and
make the analytics modules benefit from improvements to the MC engine.

3. Move __main__ demo scripts out of library modules

Having if __name__ == "__main__": blocks inside fbsabr.py, mcheston.py, black.py, etc. conflates
library code with runnable examples. A standard approach is an examples/ folder with standalone
scripts. It also removes the bare import black / import bachelier bugs (issue #5 in the review)
as a side effect.

4. Define protocols for market data boundaries

eqvolsurface.eqvolsurfacedata_from_file() returns an object whose shape is implicit — consumers
must know that it has .expiries, .forwards, .vols, .get_strikes(). A typing.Protocol
(or a @dataclass) would make the contract explicit and allow static analysis to catch misuse.
The same applies to the parameters dictionaries passed to the MC pricers
({'LnVol': ..., 'Beta': ..., 'Rho': ...}) — these should be typed dataclasses, not plain dicts.

5. Dependency injection for the calibration solver

The optimizer used in calibration is currently hardcoded inside each subclass. Passing the solver
in (e.g. calibrate(date, options, optimizer=LevenbergMarquardt())) would make the calibration
testable, swappable, and independent of the surface definition.

Realistically, items 1 and 2 are the most impactful and the hardest. Items 3–5 are straightforward
cleanups. Completing all five would justify an 8.5 or 9.
